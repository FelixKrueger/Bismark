# Plan Review A — `bismark-methylation-consistency`

**Reviewer:** A (independent, fresh context)
**Date:** 2026-05-29
**Inputs reviewed:** `SPEC.md`, `PLAN.md` (same dir); Perl `methylation_consistency` (repo root); `bismark-io` src (`lib.rs`, `read.rs`, `write.rs`, `record.rs`, `tags.rs`, `pair.rs`); `bismark-dedup` src (`main.rs`, `cli.rs`, `filename.rs`, `report.rs`, `lib.rs`, `pipeline.rs`) + its `tests/byte_identity_real_data.rs`; `bismark-extractor/src/logging.rs`.

**Verdict:** The plan is **fundamentally sound and unusually well-cross-checked against the Perl source.** Most byte-identity claims hold up against the source. There are **3 Critical** findings (one is a genuine correctness gap in the acceptance gate; two are Perl-semantics edge cases that, if implemented as the SPEC currently reads, will diverge), plus several Important and Optional items. None of the Criticals require re-architecting; they require tightening the contract before implementation.

## Logic review

### L1 (CRITICAL) — The byte-identity acceptance gate's BAM comparison does NOT match the precedent it claims to mirror
SPEC §7.2 / PLAN D1 say the three output BAMs are compared "at the decompressed record level … same set+order of records per bucket … ordered (qname, flag, rname, pos, mapq, cigar, seq, qual, **all tags**)". The plan repeatedly states this "mirrors `bismark-dedup`'s precedent."

It does **not**. The dedup real-data gate (`tests/byte_identity_real_data.rs:188-200, 254-292`) compares only a **`HashSet<String>` of retained qnames** — unordered, qname-only, no FLAG/POS/CIGAR/SEQ/QUAL/tags. dedup gets away with that because dedup's *output ordering equals input ordering by construction* and its risk is "which reads survive," not "are the bytes right." methcons has a **different and stronger** risk profile:
- methcons is a **pure pass-through splitter** — every output record's bytes (incl. all tags, SEQ, QUAL, the XM string itself) must round-trip through noodles parse→serialize unchanged. The header is *re-serialized*, not byte-copied (SPEC §8.3). A qname-set comparison would pass even if noodles silently dropped/reordered an optional tag, rewrote a SEQ, or mangled a tag's integer subtype on every single record.
- Order matters for PE (R1 must immediately precede R2) and the SPEC explicitly asserts order — but a `HashSet` discards it.

**This is the single highest-risk silent-wrong-output path in the whole port and the proposed gate as-mirrored-from-dedup would not catch it.** The SPEC's *prose* (full ordered tuple incl. all tags) is correct; the PLAN must make the test *actually do that* and must stop claiming dedup's qname-set method is the precedent to copy. Recommended: read both Perl and Rust outputs back with `bismark_io::open_reader` and assert an **ordered `Vec` of (qname, flag, rname, pos, mapq, cigar, seq, qual, sorted-tag-list)** is equal — OR shell out to `samtools view` on both and `assert_eq!` the full text (simpler, and sidesteps any noodles-reads-its-own-output blind spot — see L2).

### L2 (IMPORTANT) — Reading Rust output back *through the same noodles reader that wrote it* can mask round-trip bugs
If the verification reads both files via `bismark_io::open_reader` and compares the resulting `BismarkRecord`s, any *symmetric* noodles encode/decode bug (e.g. a tag subtype that's written wrong but read back to the same wrong value) is invisible. The Perl baseline is produced by `samtools`, so the cross-check against Perl partially mitigates this — but only for fields the comparison actually inspects, and `BismarkRecord` construction itself *drops* records the Perl kept (see L4). Strong recommendation: at least one gate (the Phase D real-data one) should compare via **`samtools view` text on both** rather than noodles-vs-noodles. The dedup suite already shells out to the Perl binary, so adding `samtools view` is in-keeping.

### L3 (CRITICAL) — Round-then-compare: the SPEC's own example is arithmetically impossible, which signals the boundary semantics are not yet pinned
SPEC §2.5's load-bearing claim is correct in *spirit* and matches Perl `methylation_consistency:266,272,282`: Perl computes `sprintf("%.1f", meth/(meth+unmeth)*100)` and then compares that **string** numerically (`$percent_methylated <= $lower_threshold`). The PLAN's classify (A3 step 3-4: `format!("{:.1}", …).parse::<f64>()` then compare) is the right transcription. **But the worked example is wrong and exposes an unverified assumption:**

> "a raw 10.04 % read rounds to `"10.0"` → unmethylated, while 10.05 % rounds to `"10.1"` → mixed"

`10.05` is **not representable** — the nearest f64 is `10.0499999999999989…`, which both Perl's `sprintf` and Rust's `{:.1}` round to `"10.0"`, not `"10.1"`. More importantly, **no real `meth/total*100` ratio lands on `*.?5` exactly** except where `total` is a power of 2 (e.g. `1/8*100 = 12.5`, `3/8*100 = 37.5`), and *those* are the only places round-half-to-even ("banker's rounding", which BOTH C `printf` and Rust use) could bite. The SPEC hand-waves this as "exact-halfway ties are essentially unreachable" (§8.1, confidence "high") — that is **false**: `total ∈ {2,4,8,16,…}` is common (a read with exactly 8 CpG calls is ordinary), so `12.5`, `37.5`, `62.5`, `87.5` ARE reachable and ARE exact halves. The good news: Perl `sprintf` and Rust `{:.N}` use the **same** round-half-to-even on the **same** stored f64, so they should still agree — but the confidence rationale is wrong, and the example is wrong. **Fix the SPEC's example and rationale, and make Spike 1 explicitly include the power-of-2 `total` halfway cases** (`12.5/37.5/62.5/87.5/...` at `.1`, and `*.??5` cases at `.2` for the report). This is cheap and turns a hand-wave into a proof. (See also A1 under Validation.)

### L4 (IMPORTANT) — "Stricter validation is a no-op on genuine data" is unverified for the *acceptance* datasets and can flip the gate from pass to error
SPEC §4.1 accepts that `BismarkRecord::from_noodles_record` (`record.rs:116-141`) errors on any record missing `XR`/`XG` or with `XM.len() != seq.len()`, where Perl reads **only `XM`** (`methylation_consistency:209`). Verified: the Rust constructor hard-errors (`tags::xr`/`xg` → `MissingTag`; the length check → `XmSeqLengthMismatch`). The plan's bet is "genuine Bismark BAMs always carry all three with matching lengths." That is *true for standard Bismark output*, but the byte-identity gate runs Perl and Rust on the **same** file and demands identical output. If the real 10M datasets contain *even one* record where `XM.len() != seq.len()` (possible after some trimming/soft-clip workflows, or for the experimental `--chh` data, or any third-party-processed BAM), Perl **succeeds** and emits it; Rust **aborts the whole file** with an error → gate fails not because of a logic bug but because of the strictness divergence. Recommendation: keep `BismarkRecord` for max reuse (agree with the plan), but (a) Phase D must *measure* this on the actual datasets before declaring the gate green, and (b) decide *now* whether a single malformed record should abort (current behaviour) or be skipped-with-warning — the Perl does neither cleanly, so document the chosen divergence loudly. Do not leave this as "should never happen."

### L5 (CRITICAL) — Perl's missing-XM behaviour writes a TRUNCATED file, and the SPEC's transcription is subtly wrong about WHICH records survive
SPEC §2.5 step 1 and §4.6 describe Perl's `last` on a missing-XM R1 (`methylation_consistency:224-227`) as "stop processing this file entirely, then fall through to the summary with whatever was accumulated." That is correct. But there are two under-specified consequences the PLAN must nail, because they directly determine byte-identity:

1. **The output BAMs are truncated at the bad record, not failed.** Perl `last`s out of the `while` loop, closes the output handles, and writes a *report reflecting only the records seen so far*. So on a file whose 500th record lacks XM, Perl emits 3 BAMs containing only buckets from records 1-499 and a report totalling 499. **The Rust port must reproduce this "stop-and-finalise-partial-output" behaviour, NOT error out.** PLAN C2 says "fatal (truncation, mate mismatch) → error out" and lumps missing-XM under the same heading in places — but missing-XM is **not fatal in Perl**; it's a graceful early stop. If Rust treats a missing-XM record as a hard error (which is what `BismarkRecord::from_noodles_record` would do — it returns `MissingTag{tag:"XM"}`), the output files won't match Perl's partial output. **This interacts with L4**: because `.records()` surfaces the missing-XM as an `Err`, the pipeline must catch that specific error and convert it to "stop, finalise, write report" rather than propagate.
2. **`min_count==0` "zero gate"** (`methylation_consistency:265,269`, SPEC §2.5 step 5): a read with `meth+unmeth==0` AND `min_count==0` is `next`'d → counted in **no** bucket and **not** discarded — so `total` in the report can be *less* than the number of records read. The PLAN captures this (A3 `Routing::Skip`, gotcha §8.8) — good — but note it must NOT be conflated with the missing-XM stop. Keep them as distinct outcomes.

The PLAN should add an explicit Phase-C test: "BAM whose Nth record lacks XM → 3 BAMs contain only records 1..N-1, report totals reflect 1..N-1, exit code 0 (Perl doesn't die)."

### L6 (IMPORTANT) — `bam_isTruncated` divergence is acceptable but the exit-code contract is unstated
Perl `bam_isTruncated` (`:422-443`) `die`s (nonzero exit) on a truncated BAM. noodles surfaces truncation as an I/O error mid-stream. The plan maps this to "a clear error" (SPEC §4 / PLAN C2). Fine — but note the *timing* differs: Perl checks truncation **up front** (before opening outputs, `:154-156`), so on truncation Perl writes **no** output files at all. Rust, streaming via noodles, may have already created+written several bucket BAMs before hitting the truncation error mid-stream → leftover partial files. Since truncation-error *text* is explicitly out of the gate (§7), this is probably acceptable, but the **leftover-partial-files** side effect should be documented, and ideally the Rust port should clean up (or the test should not assert on their absence). Low stakes, but call it out.

### L7 (OK, verified) — SE/PE auto-detect `None → SE` is correct and does NOT inherit dedup's error
Verified against Perl `:354-393` + `:157-177`: when no Bismark `@PG` line matches, `$single`/`$paired` are returned **undefined**; back in `process_file` neither `if($single)` nor `elsif($paired)` fires, so `$paired` stays falsy → `$type = "single-end"` and the PE-only `test_positional_sorting` is skipped. So `None ⇒ SE`, silently, **not** an error. The PLAN (B1) maps `None ⇒ SE` correctly. Critically, this is the **opposite** of `bismark-dedup`, whose `resolve_paired_mode` (`main.rs:376-378`) returns `Err(CannotAutoDetectMode)` on `None`. The plan flags this divergence (SPEC §3 table note, §4) — good — but because the dedup code is the literal mirror source, **the implementer is at high risk of copy-pasting the error path.** Make this a loud inline comment in the eventual `resolve_mode` and a dedicated unit test (PLAN B4 has it: "no Bismark `@PG` ⇒ SE test" — good, keep it and make it assert exit 0 + `single-end` in the report).

### L8 (IMPORTANT) — `bam_isEmpty` ordering vs `open_reader`'s eager sort-check + header read
Perl's order is: `bam_isEmpty` (skip-and-return if zero records) → `bam_isTruncated` → mode-detect → (PE) `test_positional_sorting` → open outputs. The Rust plan (A7) opens the reader first (which eagerly reads the header) then peeks the first record for emptiness. Two consequences:
- For an **empty** file, Perl `return`s and writes **nothing** and continues to the next input. The Rust port must replicate: peek → if no first record, skip file, write **no** report and **no** BAMs, continue. PLAN A7 step 2 + C2 say this ("skip file, no outputs"). Verify the report is **also** skipped — Perl opens REPORT *after* the empty check (`:199` is after the `return` at `:146`), so an empty file produces **no `_consistency_report.txt` either**. The SPEC/PLAN should state explicitly that the empty case produces **zero** files including the report (easy to accidentally still emit an all-`N/A` report).
- A **coordinate-sorted SE** file: Perl never sort-checks SE, so it processes normally. Rust must use the `without_sort_check` path for SE (see A2). But note `open_reader` *also* reads the header eagerly and would reject coordinate sort *before* the empty-peek. So the empty-check-then-skip can't even be reached for a coordinate-sorted file unless the no-sort-check reader is used. Sequencing matters — see A2/Open-Decision-1.

## Assumptions

### A1 (CRITICAL→resolve before impl) — Open Decision #1: SE needs a no-sort-check path, and the recommended `bismark-io` addition is the right call but the plan understates what's already available
Verified: `open_reader` (`read.rs:562-575`) → `BamReader::from_path` → `BamReader::new` → `check_not_coordinate_sorted` **always** (`read.rs:234-239`). There is **no** `without_sort_check` reachable through `open_reader`. So the plan's premise is correct: SE-correct behaviour (process coordinate-sorted SE files) is **not** reachable via `open_reader` today.

However — `bismark-io` **already exposes** `BamReader::without_sort_check`, `SamReader::without_sort_check`, `CramReader::from_path_without_sort_check`, and `ThreadedBamReader::from_path_without_sort_check` (read.rs:244, 396, 454, 332). What's missing is only the **path-dispatching** convenience (`open_reader_without_sort_check`). The plan's Option (a) ("reimplement a tiny format-detect + `*_without_sort_check`") is wholly doable today *without touching bismark-io* by calling `AlignmentKind::from_path` then the matching concrete `without_sort_check` ctor — that's ~10 lines and zero cross-crate coordination. The plan's recommended Option (b) (add `open_reader_without_sort_check` to bismark-io) is cleaner and symmetric, but it's a **separate crate's API change** that drags in its own review/version-bump cycle and is on the critical path of Phase A.

**Reviewer A recommendation:** Do **Option (a)** for v1.0 (self-contained, unblocks Phase A immediately, uses only already-public ctors), and file Option (b) as a *follow-up* bismark-io ergonomics PR. Do not block the methcons MVP on a bismark-io release. Either way: methcons should call the no-sort-check path for **both** SE and (the Perl-equivalent of) PE-after-its-own-`@HD SO` guard — see A3. *(Note the subtlety: the plan wants SE = no sort check, PE = sort check. Since methcons resolves SE/PE from the header which requires the reader already open, and `open_reader` rejects coordinate sort eagerly, you must open with `without_sort_check` **unconditionally** and then re-implement the PE-only `@HD SO:coordinate` rejection yourself — PLAN B2 already plans this. So in practice the port should ALWAYS open no-sort-check and gate the coordinate rejection on `is_paired`. Make that explicit; the current A7/B2 split reads as if SE and PE use different openers.)*

### A2 (IMPORTANT) — Open Decision #5 (binary name): agree, but verify the drop-in story
Plan recommends `methylation_consistency_rs` (Perl name + `_rs`), matching dedup's `deduplicate_bismark_rs`. Verified dedup uses exactly that (`cli.rs:44`). Extractor uses a hyphen style. **Recommend `methylation_consistency_rs`** — consistency with dedup (the literal mirror) > consistency with extractor, and underscore-preserving matches the Perl invocation name char-for-char. Decision is low-stakes and reversible.

### A3 (IMPORTANT) — Open Decision #3 (PE pairing): agree with manual qname-equality, but the SPEC must state the FLAG-bit divergence is intentional
Verified Perl `:238-239` compares only `$id1 eq $id2` (raw qname, no FLAG inspection, no `/1`/`/2` stripping in the *main loop* — the stripping only happens in the dropped `test_positional_sorting` pre-flight at `:493-497`). `BismarkPair::from_mates` (`pair.rs:40-78`) additionally requires `r1.read_identity()==R1` and `r2.read_identity()==R2`, which Perl does **not** check. So `from_mates` is **stricter** than Perl and would `die` on (e.g.) a pair where both mates carry the same FLAG, where Perl would happily process. **Recommend manual qname-equality** (PLAN B3) for faithfulness — agreed. But note a *second* subtlety the plan misses: Perl's main-loop qname check does **NOT** strip `/1`,`/2` suffixes, while the dropped pre-flight *did*. Modern Bismark doesn't append `/1`,`/2`, so this is moot for genuine data — but if a legacy file has `read/1` and `read/2`, Perl's **main loop** would `die` (`"READ IDs … did not match"`) even though the file is validly paired. The Rust port reproducing the main-loop check faithfully will also die → byte-identical (both fail). Fine, but document that legacy `/1`,`/2`-suffixed files are an error in BOTH.

### A4 (OK) — Open Decision #4 (`--samtools_path` accept-and-ignore): agree
Perl uses `$samtools_path` for all I/O and validates existence (`:94-123`). Since noodles does all I/O, the path is genuinely unused. dedup already set the accept-and-ignore precedent (`cli.rs:120-121, 230`). Validating existence would *add* a failure mode Perl has but that has **no bearing on output** and could spuriously fail the gate on a machine where the path is wrong but noodles works fine. **Recommend accept-and-ignore** (agreed). One caveat: Perl `die`s if `--samtools_path /bad/path` doesn't exist — so a user script relying on that error won't get it. Acceptable; note in §4.

### A5 (IMPORTANT) — `min-count` validation: Perl allows `0` AND the Rust `u32` type silently diverges on negative input message
Verified Perl `:55` validates `$min_count =~ /^\d+$/` → `0` is allowed, negatives/non-numeric `die`. The SPEC (§2.2, §6.1) captures "0 allowed" correctly. But: clap `u32` parsing rejects `-1` with clap's **own** error message and exit code 2, while Perl's is `"Min count must be a positive integer, not '-1'\n"` and exit nonzero (die→255). Since CLI error *text* is out of the gate (§7), this is acceptable — but the SPEC says "Negative/non-numeric → clap parse error" which is right. Just confirm the **exit code** expectation in tests isn't asserting Perl's 255 vs clap's 2. Also note Perl's regex `^\d+$` would **reject** `+5` and `5.0` and leading/trailing whitespace — clap `u32` also rejects these, so behaviourally aligned. Minor.

### A6 (OK, verified) — Report format strings
Cross-checked SPEC §5.1 against Perl `:334-343` byte-by-byte:
- `"Total $type records     -\t…"` — 5 spaces before `-`. ✓ (SPEC §5.1 line 1 says "5 spaces").
- Separator: counted the hyphens in Perl `:335` — **49** hyphens. ✓
- `"All methylated    [ >= $upper% ] -\t…"` — `All methylated` + 4 spaces. ✓
- `"All unmethylated  [ <= $lower% ] -\t…"` — `All unmethylated` + 2 spaces. ✓
- `"Mixed methylation [ ${lower}-${upper}% ] -\t…"` — `Mixed methylation` + 1 space. ✓
- `"Too few CpGs   [min-count $min] -\t…"` — `Too few CpGs` + 3 spaces; CHH variant `"Too few CHHs   "` + 3 spaces. ✓
The SPEC's instruction to **copy the literal format strings out of the Perl source, not retype** (PLAN A5) is the right safeguard. **Strongly endorse** — and the test should pin the EXACT expected bytes for at least one full report (dedup's `report.rs` tests do exactly this with `const EXPECTED_TYPICAL`). One gotcha the SPEC should add explicitly: the report has **NO leading newline** (unlike dedup's report which starts with `\n`). Line 1 is `Total …` with no preceding blank line. Easy to get wrong by reflex-copying dedup's `\nTotal…`.

### A7 (IMPORTANT) — `BamWriter` eager-header + lazy-bucket interaction
Verified `BamWriter::new` writes the header **eagerly at construction** (`write.rs:59-63`) and `from_path` creates the file immediately (`write.rs:48-49`). So the plan's "lazy writers — create on first record" (A7 step 4) means: a bucket file is **created on disk the moment its first record arrives**, with the header written then. This correctly reproduces Perl's lazy-header-on-first-record (`:273-275` etc.) for **non-empty** buckets. Good. The only open question is the **empty** bucket — see A8 / Open Decision #6.

### A8 (CRITICAL→must resolve via Spike 2 before Phase C, agree) — Empty-bucket BAM is genuinely ambiguous and "lazy → no file" may FAIL the gate
Open Decision #6 / Spike 2. Perl opens the samtools pipe **eagerly** (`:196-198`, `open(METH,"| samtools view -b -S - > file")`), so the file is created at process start regardless of records; the SAM header is printed only on the first record (`:273-275`). For an **empty bucket**, Perl pipes *nothing* into `samtools view -b -S -` → samtools reads empty stdin → emits… something (likely a valid-but-headerless BGZF EOF block, i.e. a ~28-byte file, OR an error to stderr, depending on samtools version). **So Perl almost certainly produces a (tiny, possibly header-less) FILE for an empty bucket, while the plan's "lazy → no file" produces NO file.** If the gate asserts "same set of output files exist," lazy-no-file **fails**. The plan correctly defers this to Spike 2 and flags it as out-of-gate "(pending Spike 2)" in §7 — that's the right call. **Reviewer A position:** Spike 2 is correctly scoped and MUST run *before* the gate is declared, because the answer changes the acceptance criteria. Do NOT pre-commit to "lazy → no file" in the SPEC (it's currently phrased as the recommended default — soften to "pending Spike 2; likely needs a header-only/empty file to match"). My prior: Perl emits a tiny file, so methcons will probably need to **eagerly create + finalise** at least a header-only BAM for empty buckets to match — which conflicts with the lazy design. Resolve early.

## Efficiency analysis

### E1 (OK) — Streaming, single-pass, O(records) time, O(1) memory (modulo 3 open writers). Correct for the workload.
Each record: one `XM` slice scan (`count_xm`), one classify (constant), one write. No buffering of records, no sort, no global state beyond 4 counters + 3 `Option<BamWriter>`. This is the right shape and matches Perl's streaming model. The `format!("{:.1}",…).parse::<f64>()` per-record (A3 step 3) allocates a tiny String per record — at 10-50M records that's 10-50M transient allocations. **Optional micro-opt:** since you only need the rounded value for a `<=`/`>=` comparison, you could compute the round-half-to-even tenth without formatting (`(x*10.0).round_ties_even()/10.0`) — BUT that risks diverging from `sprintf` semantics at the exact boundaries, which is the whole load-bearing concern. **Recommendation: keep the format!/parse approach** (correctness > a sub-percent alloc cost; the workload is I/O-bound on BGZF anyway, as the dedup/extractor profiling already established). Just don't let a reviewer "optimize" it away later — add a comment.

### E2 (OK) — v1.0 single-threaded is the right scope
Matches dedup v1.0 and the SPEC §9 deferral. Given methcons is I/O-bound (decompress BAM, recompress 3 BAMs), the future win is `ThreadedBamReader`/`ThreadedBamWriter` (already in bismark-io). Deferring is correct.

### E3 (MINOR) — Multiple input files loop: confirm each file's writers are independently finalised
PLAN C2 "Multiple input files: loop independently." Ensure the 3 `Option<BamWriter>` are created/finished **per file** (not reused across files — each file has its own header and output paths). The lazy-writers struct must be re-instantiated per input. Trivial but a classic bug if the struct is hoisted out of the per-file loop.

## Validation sufficiency

### V1 (CRITICAL) — see L1: the BAM comparison must be ordered + all-tags, not dedup's qname-set. This is the #1 validation gap.

### V2 (IMPORTANT) — Spike 1 grid must include the actually-reachable halfway cases
Per L3: the grid must include `total ∈ {2,4,8,16,32}` (exact `.5` halves at `.1`) and the `.2`-level halves for the **report** percentages (`bucket/total*100` where `total` is the 4-bucket sum — e.g. 1/8 = 12.50, 3/8 = 37.50). The SPEC's "dense grid of (meth,total)" is good but must be *guaranteed* to hit the power-of-2 totals, not just a uniform sample that might skip them. Also: the report uses `{:.2}` and the classify uses `{:.1}` — Spike 1 must test **both** precisions, and must test the **round-then-compare decision** at thresholds (e.g. construct meth/total such that the rounded pct == lower exactly, and pct == upper exactly, to lock the `<=`/`>=` boundary inclusivity). PLAN C3 says it does the classification check — good; make the power-of-2 totals mandatory.

### V3 (IMPORTANT) — Threshold-boundary classification tests must cover the inclusive edges with the *defaults* AND custom thresholds
Perl: `<= lower` (inclusive unmeth), `>= upper` (inclusive meth), else mixed. With defaults lower=10, upper=90: a read that rounds to exactly `10.0` → unmeth; `90.0` → meth; `10.1`/`89.9` → mixed. PLAN A3 unit tests mention "pct exactly lower/upper" — ensure both the **default** and a **custom** (e.g. lower=0, upper=51) threshold pair are tested, including lower=0 (a read rounding to `0.0` → unmeth; a 0%-methylated read with enough calls → unmeth, NOT skipped, since total>0).

### V4 (IMPORTANT) — Add the missing-XM-mid-file truncation-of-output test (see L5)
No proposed test currently asserts the **partial-output-on-missing-XM** behaviour. This is a real Perl behaviour (`last`) that the strict Rust reader will naturally turn into a hard error unless explicitly handled. **Highest-risk untested path after L1.** Add: SE BAM with a mid-stream missing-XM record → assert 3 BAMs hold only the pre-bad records, report totals match, exit 0.

### V5 (IMPORTANT) — `total == 0` report path is reachable via TWO routes; test both
Perl yields `N/A` percentages when the 4-bucket sum is 0 (`:316-318`). Reachable when: (a) every record was discarded (all had `< min_count` calls) → `discarded>0` so total≠0 — wait, no: discarded **counts toward** total (`:259-262, :310`), so all-discarded gives total>0 and real percentages. (b) The file's **first record lacks XM** → `last` immediately → all 4 counters 0 → total 0 → all `N/A`. (c) `min_count==0` and every record has 0 calls → all `Skip`'d → total 0 → `N/A`. The PLAN (C2) mentions the N/A path but should enumerate (b) and (c) as the two reachable routes and test at least (c) (cleanest to construct). Note (b) overlaps with V4.

### V6 (OK) — CHH path tests (PLAN C1) are sufficient: `_CHH` filenames, `Too few CHHs` label, H/h counting. Verified the label spacing matches `:340` (`"Too few CHHs   "` = 3 trailing spaces, same as CpG variant). Good.

### V7 (MINOR) — No test asserts the header is byte-preserved through the noodles round-trip on a *real* multi-`@SQ`/`@PG`/`@CO` header
SPEC §8.3 calls this "solved by dedup" — but dedup's gate (qname-set) doesn't actually compare headers either! The only place header fidelity is truly tested is dedup's *unit* round-trip tests on a synthetic 1-`@SQ` header. For methcons, where the header is copied verbatim and is part of the gate, Phase D MUST compare the output header (via `samtools view -H` on Perl and Rust outputs) on a real multi-contig human header. Add it explicitly.

## Alternatives

### ALT1 — Bypass `BismarkRecord` for true Perl-XM-only leniency (rejected, correctly)
The plan considers iterating raw `RecordBuf`s + `tags::xm` directly (SPEC §4.1 open question), which would match Perl's "only read XM, tolerate missing XR/XG, tolerate XM/seq length mismatch." This would **eliminate L4** (the strictness-flips-gate risk) and is arguably *more* faithful for a pure pass-through splitter that never uses strand. **Trade-off:** loses the `BismarkRecord` reuse (strand classification, the validated `xm()` accessor) and adds a parallel code path. Given methcons **never uses strand** (it only counts XM bytes and passes records through), the `BismarkRecord` machinery is largely dead weight here. **Reviewer A's nuanced position:** the plan's choice (use `BismarkRecord`) is defensible for reuse/consistency, **but** L4+L5 mean the implementer must catch `MissingTag{XM}` and `XmSeqLengthMismatch`/`MissingTag{XR/XG}` errors from `.records()` and translate them to the *Perl-equivalent* outcome (missing-XM → graceful stop; XR/XG/length issues → ??). That translation is fiddly. The raw-`RecordBuf` path would make the Perl semantics **trivially** correct (just regex/scan XM, ignore everything else) at the cost of less reuse. **Recommend the plan owners genuinely re-weigh this**, because the "max reuse" argument is weakest precisely here (no strand needed) and the strictness divergence is the plan's biggest correctness risk. At minimum, if keeping `BismarkRecord`, the error-translation rules must be specified explicitly, not deferred.

### ALT2 — Use `samtools view` text-diff as the PRIMARY gate (recommended for Phase D)
Rather than noodles-read-back tuple comparison (which has the L2 self-blind-spot), shell out to `samtools view` (no `-H` for records, `-H` for header) on both Perl and Rust outputs and `assert_eq!` the text. This is what the Perl pipeline effectively produces anyway, sidesteps any noodles encode/decode symmetric bug, and is simpler to write than an ordered-tuple comparator. dedup already shells out to the Perl binary in its gate, so adding `samtools` is consistent. Keep a noodles-based unit-level check for the synthetic Phase-A/B fixtures (fast, no samtools dependency in the inner loop) but make the **real-data Phase-D gate** samtools-text-based.

### ALT3 — Pull Spike 1 before Phase A (the plan already floats this)
PLAN §"Estimated sequencing" notes Spike 1 "could be pulled forward to the start of C (or even pre-A)." **Reviewer A: pull Spike 1 to pre-A.** It de-risks the single load-bearing numerical claim (round-then-compare) that the entire classify core depends on, costs ~30 min (a Perl one-liner + a Rust snippet), and if it surprises us (it shouldn't, but the SPEC's rationale for "high confidence" is demonstrably wrong per L3), it changes `classify.rs`'s design. Spike 2 (empty-bucket) should run before Phase C as planned (it gates the file-existence acceptance criterion). **Phasing otherwise A→B→C→D is sensible** — SE-first MVP, then PE, then CHH/edges, then real-data gate. No phase is missing; the one resequencing is Spike 1 → pre-A.

## Summary
The plan is thorough and its Perl cross-references are mostly accurate (report format strings, `None⇒SE`, `min_count==0` zero-gate, PE-counts-pairs, threshold ranges all verified correct against source). The serious risks are concentrated in (1) an acceptance gate that, as written to "mirror dedup," would silently miss per-record byte corruption (dedup only compares an unordered qname set); (2) the missing-XM graceful-stop semantics colliding with the strict reader's hard-error; (3) an incorrect "halfway ties unreachable" rationale that needs the spike to actually prove parity; (4) the SE no-sort-check path; and (5) the empty-bucket file ambiguity. All are resolvable in the SPEC/PLAN before implementation — none require redesign.
