# PLAN (kickoff) — Bismark Rust suite: wiring, packaging & distribution

> **Status:** PLANNING KICKOFF — framing + decisions, not a finalized SPEC. Modeled on the **Trim Galore Rust** release machinery (`~/Github/TrimGalore`), adapted for a multi-binary suite. **No infra built yet.** Bioconda is explicitly **out of scope until a mature/GA release** (Felix).

## 1. The proven template — how Trim Galore (Rust) ships
Trim Galore is a **single crate / single binary** (`trim_galore`, version in `Cargo.toml`). Its release story (`.github/workflows/release.yml` + `justfile` + `build.rs` + `Dockerfile`):
- **Trigger:** `workflow_dispatch` (manual) with a **`dry_run`** input; version read from `Cargo.toml`; release allowed from `master` (GA) or `dev` (prerelease only, detected by a `-` in the version). The **workflow creates + pushes the tag itself** (never push `v*` manually) and asserts `Cargo.lock` == `Cargo.toml` version.
- **Prebuilt binaries:** matrix of **3 targets** — `linux-x86_64`, `linux-aarch64`, `macos-aarch64` — `cargo build --release --locked --target`, packaged as `<bin>-<target>.tar.gz` + `.sha256`, uploaded to the GitHub Release; **smoke-tested** (`--help`/`--version`) before publish.
- **Docker:** multi-arch (amd64 + arm64) built on native runners, pushed **by digest** to **GHCR**, merged into one manifest with tags `version` / `major.minor` / `major` / `latest` (GA) or `beta` (prerelease) / `dev`; smoke-tested.
- **crates.io:** `cargo publish` via OIDC (`crates-io-auth-action`) as the last step.
- **Versioning/provenance:** `build.rs` stamps the **git short-hash** + a **reproducible build epoch** (`SOURCE_DATE_EPOCH`); a `justfile` `reproduce` target diffs two builds for bit-identity. `justfile` mirrors CI locally (`fmt`/`clippy`/`test`/`test-release`).
- **Citation:** `CITATION.cff` + `.zenodo.json` (DOI on GitHub release).

## 2. Current Bismark Rust state (what we're wiring)
- **Workspace** `rust/` — **13 crates**: 1 shared lib **`bismark-io`** (`=1.0.0-beta.8`, path+exact-version pinned by every binary crate) + **12 binary crates**. `[workspace.package]` already shares edition/license/repo/authors + the release profile (lto, codegen-units=1, strip).
- **Per-crate INDEPENDENT versions** today: io `1.0.0-beta.8`, dedup `1.2.1-beta.1`, aligner `1.0.0-alpha.1`, c2c/bedgraph `1.0.0-beta.2`, the rest `1.0.0-beta.1`.
- **12 binaries, inconsistently named** with a transitional **`_rs` dev suffix** (so they coexist with the Perl tools): `bismark_rs`, `bismark_genome_preparation_rs`, `deduplicate_bismark_rs`, `bismark-methylation-extractor-rs` *(hyphens — odd one out)*, `bismark2bedGraph_rs`, `coverage2cytosine_rs`, `bismark2report_rs`, `bismark2summary_rs`, `bam2nuc_rs`, `NOMe_filtering_rs`, `filter_non_conversion_rs`, `methylation_consistency_rs`.
- **Infra present:** `rust_ci.yml` (clippy/fmt/cargo-test/perl-oracle) + repo Perl CI. **Absent:** no `justfile`, no `Dockerfile`, **no release workflow** — this is the gap.
- **Runtime externals (unlike Trim Galore):** the aligner wraps **Bowtie 2 / HISAT2 / minimap2** subprocesses and the pipeline uses **samtools** — the container/install story must account for these (Trim Galore bundled FastQC *into* its binary; Bismark cannot bundle the aligners).
- **Perl coexistence:** the Perl suite still lives at the repo root on the main branches; the Rust rewrite is on `rust/iron-chancellor`.

## 3. Proposed distribution architecture (the suite adaptation)
1. **One suite tarball per target** (not per-binary): `cargo build --release --locked` builds all 12 bins; package them **together** into `bismark-rust-<version>-<target>.tar.gz` (+ `README`/`LICENSE` + `.sha256`), 3 targets. Users download one archive → the whole suite. (Per-binary = 36 artifacts, rejected.)
2. **Docker (GHCR, multi-arch)** carrying all 12 binaries **+ samtools + the aligners** (at least Bowtie 2; HISAT2/minimap2 optional/variant images) — a "batteries-included" Bismark image. Same digest→manifest pattern as Trim Galore.
3. **`build.rs` + `justfile`** — adopt Trim Galore's verbatim (git-hash + `SOURCE_DATE_EPOCH` stamping; `justfile` for local CI parity + a `reproduce` target). Shared at the workspace level.
4. **`release.yml`** — adapt Trim Galore's: dry-run, version-from-manifest, workflow-owned tag, smoke tests. The matrix builds the **workspace** once per target; packaging + Docker carry all bins.
5. **Citation:** `CITATION.cff` + `.zenodo.json` at the mature release.

### ✅ Resolved (Felix, 2026-06-06; rev 3 = post-dual-review)
- **D1 = (c) hybrid, CLARIFIED:** the **suite version is a RELEASE-level concept** (the git tag + GH release + Docker tag + tarball name + every binary's `--version`), sourced from ONE place — a `[workspace.package] version` in `rust/Cargo.toml` (which has none today; the workspace is virtual). The **per-crate `Cargo.toml` versions stay INDEPENDENT** (dedup `1.2.x`, aligner alpha, `bismark-io` own semver — needed for the GA crates.io publish); binary crates do **NOT** blanket `version.workspace = true` (that would erase those lineages — review). So: suite version drives what *users* see; crate versions drive *crates.io* (at GA). The release workflow asserts the suite version source ⇄ tag, separately from any crate-version check.
- **D2 = DEFER ALL crates.io to GA** *(revised from "publish `bismark-io` now" after dual review — both reviewers Critical):* **NO crates.io publishing during beta.** `bismark-io` stays at its already-published `beta.6`; beta channels = binaries + Docker only. Rationale: publishing local `beta.8` would freeze (crates.io is **immutable**) an API the project's **no-bump-during-beta / additive** convention treats as mutable, and force a version-bump cascade across the **8–9 crates exact-pinning `=1.0.0-beta.8`**. ALL crates.io (bismark-io current + the 12 binary crates) clusters at GA with bioconda.
- **D3 = (a) keep `_rs`** (normalize: fix the extractor's `bismark-methylation-extractor-rs` hyphens → uniform `<tool>_rs` across all 12) and **D4 = (a) beta-preview track** coexisting with Perl. Canonical-name takeover + bioconda + full crates.io = the **2.0 GA** cut.
- **D5 = (b) batteries-included Docker** — **with PINNED aligner versions** (Bowtie 2 **2.5.5**, HISAT2 **2.2.2**, minimap2 **2.31-r1302**) + samtools 1.23.1: byte-identity holds ONLY against these pins (review Critical — a stock `apt-get install bowtie2` would ship a non-matching build). Build/install the pinned versions explicitly; document the version-sensitivity.

## 4a. Concrete deliverables (beta track — this planning unit; rev-3 review findings folded)
1. **`rust/build.rs`** — git short-hash + `SOURCE_DATE_EPOCH` reproducible-build stamping (port Trim Galore's). NB: not a "workspace-shared" file (not a Cargo concept) — each binary crate references it (or a tiny shared build-helper crate). Reproducibility caveat: the workspace pulls a **C build dep (`cc` via `libmimalloc-sys`)** that TG didn't — verify `reproduce` bit-identity still holds across it.
2. **`justfile`** — local CI parity (`fmt`/`clippy`/`test`/`test-release`) + a `reproduce` bit-identity target.
3. **Unified `--version` across all 12 bins** — today each disables clap auto-version + hand-rolls `version_string()` from its own `CARGO_PKG_VERSION` in differing formats → would print numbers matching no release tag. Wire a **shared version string = suite-version + git-hash** (via `build.rs`) so every bin reports the suite version.
4. **`Dockerfile`** (batteries-included) — 2-stage; all 12 bins + samtools 1.23.1 + the **pinned** Bowtie 2 2.5.5 / HISAT2 2.2.2 / minimap2 2.31-r1302; GHCR labels. Smoke test must verify the bundled externals resolve (not just `bismark_rs --help`).
5. **`.github/workflows/release.yml`** — adapt Trim Galore's: `workflow_dispatch` + `dry_run`; **suite version from the single `[workspace.package] version` source** (NOT TG's single-crate `^version` grep — that doesn't port to a virtual workspace); a version⇄tag assertion that works for the workspace; **workflow-owned tag** with a **distinct suite scheme `bismark-rust-v<ver>`** (avoids colliding with the Perl `v0.25.1` tags + the per-crate `bismark-io-v*` tags); **branch guard** = beta **prereleases only** from `rust/iron-chancellor` (version carries a `-beta`/`-rc` suffix → TG's prerelease path; GA from `master` later); **one suite tarball per target** (all 12 `_rs` bins + `LICENSE` + `THIRD-PARTY-NOTICES` + sha256) × {linux-x86_64, linux-aarch64, macos-aarch64}; multi-arch GHCR image; **smoke-test ALL 12 bins** (`--help`/`--version`), not one; **NO crates.io job** (D2 deferred).
6. **`_rs` name normalization** — uniform `<tool>_rs` `[[bin]]` names.
7. **Licensing** — the repo's license file is `license.txt` (lowercase) → TG's `cp LICENSE` would silently no-op. Ship the GPL-3.0 text (as `LICENSE`) + a **`THIRD-PARTY-NOTICES`** for the bundled aligners (Bowtie 2 GPL, samtools MIT, HISAT2, minimap2) in every tarball + image.
- **Deferred to GA:** `CITATION.cff` + `.zenodo.json`, canonical (non-`_rs`) names + the Perl-coexistence rename mechanics (see §6), bioconda recipe, **all** crates.io publishing (bismark-io + the 12 binary crates).

## 6. Release lifecycle — the Trim Galore model (staged takeover, NOT permanent coexistence)
Resolved with Felix 2026-06-06. The byte-identical Rust port is *the same tool reimplemented*, so it succeeds Perl rather than competing with it:
- **Beta (now):** the **`bismark-rust`** track — `_rs` binaries, beta versions — **coexists** with Perl Bismark v0.25.x (A/B validation + trust-building). Distribution = prebuilt binaries + GHCR Docker + `bismark-io`→crates.io.
- **GA = Bismark 2.0.0:** rename `_rs` → **canonical names** (`bismark`, `deduplicate_bismark`, …); `bismark-rust` becomes **`bismark`**, version **2.0.0** (a major bump signals the reimplementation; byte-identity makes it a drop-in). Perl Bismark v0.25.x is **frozen** — a final tagged legacy release kept for reproducibility/legacy pipelines, not deleted, not developed. bioconda + full crates.io (the 12 binary crates) + Zenodo DOI land here.
  - **Rename-collision mechanism (review):** the canonical names collide with same-named Perl binaries. The transition is a **replacement, not a co-install** — at GA the Rust suite *is* `bismark`; a user installs *either* Bismark 2.x (Rust) *or* the frozen Perl v0.25.x, not both on one PATH. Mechanism: the repo default becomes the Rust suite; the Perl line moves to a `v0.25.x` tag (+ a `legacy/` location — reconcile with the existing `legacy/` reference already in `rust/README.md`). The beta `_rs` suffix exists precisely so the two CAN co-install *during* beta for A/B validation; the suffix drop is the deliberate end of co-existence.
- Precedent: Trim Galore did exactly this (Rust `trim_galore` 2.2.0 supersedes Perl 0.6.x).

### Resolved OQs (Felix, 2026-06-06)

- **OQ-B — release cut from `rust/iron-chancellor`** (or a release branch off it); Perl release machinery stays separate until GA.
- **OQ-C — `bismark-io` is already public on crates.io** (`1.0.0-beta.6`, 2026-05-26) — local is `1.0.0-beta.8` (2 ahead). So D2 = **keep it current** (publish the catch-up; `release.yml` automates ongoing `bismark-io` publishes). It's already crates.io-ready (description/readme/license/keywords/categories present). NB: crates.io versions are immutable — the no-bump-during-beta convention + the exact-pin to dependents means coordinate any version bump with the workspace.
