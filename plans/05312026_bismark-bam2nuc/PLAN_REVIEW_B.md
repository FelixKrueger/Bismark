# PLAN_REVIEW_B — `bismark-bam2nuc` (Rust port of Perl `bam2nuc`)

**Reviewer:** B (independent, fresh context)
**Date:** 2026-05-31
**Verdict:** **APPROVE WITH CHANGES.** The plan is well-structured, faithfully replicates the Perl quirks I could verify (the PE `or 163` bug, the substr saturation, the no-dot output-name anchor, the bytewise cache sort, the round-half-to-even rounding), and the cribbing from c2c/extractor/bismark-io is sound. There are a handful of **Critical** correctness gaps that must be closed before implementation, plus several Important refinements. None invalidate the overall design.

Method note: I read the Perl source in full, the SPEC, the PLAN, and the four sibling crates. I independently re-ran the Perl interpreter to verify every byte-identity-critical claim (PE branch precedence, revcomp on N/IUPAC, substr saturation/undef/negative, empty-field interpolation, division-by-zero, sort order, output-name quirk) and compared Rust vs Perl `%.2f`/`%.3f` rounding on half-way ties. Line numbers below are `bam2nuc:NNN`.

## Logic review

### Verified-correct claims (high confidence)

- **PE latent bug (Task C4 / SPEC §7.5 / D3).** CONFIRMED. `elsif ($flag == 83 or 163)` (`bam2nuc:226`) — Perl parses this as `($flag == 83) or 163`; the bare `163` is a truthy constant, so the elsif is **always true** and `else{die}` (`:231-232`) is dead code. I ran it for flags {99,147,83,163,0,16,256,4}: every flag ≠ 99/147 took the revcomp branch, none died. The PLAN's `correct_pe` (`if flag==99||147 {span} else {revcomp}`, never errors) is exactly right. C4's test vector `flag 0 (PE) → revcomp` and `flag 256 (PE) → revcomp` are correct.

- **SE branch (Task C4).** `calc_single_end` (`:237-252`): flag 0 → as-is, flag 16 → revcomp, else → reachable `die` (`:248-249`). The `if/elsif/else` here uses real `==` comparisons (no truthy-constant trap), so the die IS reachable. PLAN's `correct_se` returning `Err(InvalidSeFlag)` is faithful.

- **revcomp on N/IUPAC (Task C4).** CONFIRMED via Perl. `reverse` + `tr/GATC/CTAG/`:
  - `ACGTN` → `NACGT` ✓ (PLAN's vector is right)
  - `GATC` → `GATC` ✓
  - `AACG` → `CGTT` ✓ (PLAN's asymmetric vector is right)
  - IUPAC: `ARGY` → `YCRT` (R,Y untouched by tr; only GATC mapped). N and all non-GATC bytes pass through unchanged. PLAN's `match { ... x=>x }` fall-through is correct. **One missing test vector worth adding:** a span containing an IUPAC base on a flag-16/83 read, to prove the IUPAC byte survives revcomp untouched — the current C4 vectors only test N, not R/Y.

- **CIGAR `[IDSN]` skip (Task C2 / SPEC §7.3).** The Perl regex `$cigar =~ /[IDSN]/` (`:126`) matches the **letters** I, D, S, N anywhere in the CIGAR string. The PLAN's structured op-kind set {Insertion, Deletion, SoftClip, Skip} is the exact equivalent; H/P/=/X are correctly kept. The C2 test for `*` (empty CIGAR) → keep is correct: an empty/`*` cigar has no I/D/S/N letter so Perl does not skip it (and then `length$sequence` is used for the span regardless of cigar). Good.

- **Bytewise cache sort (Task B2 / B3 / SPEC §6.2).** CONFIRMED. Perl `sort keys` is bytewise on ASCII. I verified the 20-line pure-ACGTN order matches SPEC §6.2 exactly, and that a 1-char word sorts before its 2-char extensions (`A` < `AA` < `AC`). Rust `Vec<u8>::sort_unstable()` is bytewise lexicographic → identical. **IUPAC interleave nuance the PLAN under-specifies:** uppercase ASCII codes are `A`=65 … `N`=78 … `R`=82 … `T`=84. So a single-char `R` (82) sorts **after** `N`(78) and after `G`(71)<`GA`…<`GT`, i.e. between the `G*` group and `T`. A 1-char `R` sorts before `RA`/`RR`. B2's claim "verify an IUPAC key (`R`,`AR`) sorts in correctly" is right but the example placement should be pinned in the test: `AR` lands between `AG` and `AT` (A=65,G=71,R=82,T=84 → AG<AR<AT). The IUPAC golden cell (E2 cell 2) covers this empirically, which is the real safety net.

- **`%.2f`/`%.3f` rounding (Task D2 / F1).** I compiled a Rust program and compared `{:.2}`/`{:.1}` against Perl `sprintf` on exact half-way ties (6.25, 18.75, 31.25, 43.75, 0.125, 0.375). **Perfect agreement on macOS** — both round-half-to-even (6.25→6.2, 18.75→18.8, 0.125→0.12, 0.375→0.38). The PLAN's bet that no custom `%g` machinery is needed (unlike the bedGraph `%.15g` port) is well-founded: `%.2f`/`%.3f` are fixed-precision decimal, which both Rust core and glibc/macOS-libc implement as correctly-rounded round-half-to-even. The residual risk (different libc on Linux/oxy) is real but small and **F1 covers it on the target platform**. This is the correct level of caution.

- **Output-name quirk (Task D1 / SPEC §9 / D4).** CONFIRMED. `s/(bam|cram)$/nucleotide_stats.txt/` with no dot anchor (`:150`): `sample.bam`→`sample.nucleotide_stats.txt`, `foosubbam`→`foosubnucleotide_stats.txt`, `a.bam.bam`→`a.bam.nucleotide_stats.txt` (only the trailing `bam` stripped), `z.sam`→die, `noext`→die. **Additional finding the PLAN/SPEC miss: the regex is case-SENSITIVE.** `weird.BAM` → **die** (does NOT match `(bam|cram)$`). This matters for the E1 input-format gate: `AlignmentKind::from_extension` lowercases the extension (`x.BAM` → Bam, per `read.rs:80-81`), so a `.BAM` file that the Rust gate *accepts* would have its output name *fail* derivation. The Perl path-dispatch reads it (plain open) but then dies at name derivation. The PLAN's E1 gate accepts `.BAM`/`.bam` uniformly; D1 must replicate the case-sensitive strip, meaning a `.BAM` input would be accepted by the gate but rejected by `derive_output_name` (Err(NotBamOrCram)) — which actually matches Perl's net "die on `.BAM`". **But the PLAN's D1 GREEN says "if ends with `bam` or `cram`"** without specifying case — implementers could easily make it case-insensitive and diverge. Pin lowercase-exact in a D1 test (`x.BAM → Err`).

### Logic gaps / inconsistencies

1. **(Critical) `noodles-bam` is not in the dependency list.** Task A1 (line 82) enumerates deps: `bismark-io`, `clap`, `thiserror`, `noodles-fasta`, `flate2`, `noodles-sam`, `mimalloc` — **no `noodles-bam`.** But OI-1/C5 resolve the raw-record path to "either `noodles_bam::io::Reader` directly". I verified `noodles_bam::io::Reader::record_bufs(&header)` (noodles-bam 0.89.0, `reader.rs:287`) yields raw `RecordBuf`s — this is the clean path that bypasses `BismarkRecord`. The extractor does NOT declare `noodles-bam` directly (it goes through `bismark-io`'s `BamReader`). So bam2nuc must add `noodles-bam = "=0.89.0"` (matching bismark-io's transitive pin) AND `noodles-bgzf` (already a dev-dep, but the runtime BAM reader needs it for BGZF decode — `noodles_bam::io::Reader<noodles_bgzf::io::Reader<R>>`). **The PLAN's dep list cannot build the C5 reader as written.** Fix A1's deps.

2. **(Critical) Mapped record with `alignment_start() == None`.** `extract_span` (C3) takes `pos1: usize` — but the C5 driver must first unwrap `record.alignment_start()` (an `Option<Position>`). For a record that passed the raw-read path (no unmapped filter), an unmapped or malformed record could have `alignment_start == None`. The PLAN does not say what to do. Perl always has a `$start` field from `split` (column 4), which for an unmapped read is `0` → `substr($chr, -1, len)` → the **negative-offset path** (I verified `substr("ACGTACGT", -1, 4)` returns `"T"` — Perl indexes from the end!). The PLAN's `extract_span` does `pos1.saturating_sub(1)` which for `pos1=0` gives `0`, NOT the from-the-end behavior. **However**, I verified noodles `Position` is `NonZero<usize>` (noodles-core `position.rs:15`), so a record WITH `alignment_start` always has POS ≥ 1 → `saturating_sub(1) ≥ 0`, never negative → the divergence is **unreachable for any record that has an alignment_start.** The real gap is the `None` case: the PLAN must decide (skip? error? treat as the unmapped-read policy). Since real Bismark BAMs have no unmapped reads and SE unmapped reads carry flag 4 (→ `correct_se` errors anyway), the cleanest faithful choice is: do NOT special-case `None` — but the PLAN must state it and a fixture must prove it. Add a one-line decision to C3/C5: "if `alignment_start().is_none()`, the SE flag-check (`correct_se`) errors on flag 4; PE revcomps an empty span — document, do not silently skip." Right now C3's signature `pos1: usize` hides the `Option` unwrap entirely.

3. **(Important) `substr` past-end returns undef vs empty — confirm it's harmless.** I verified Perl: `substr("ACGTACGT", 8, 2)` (start exactly at len) → `""`; `substr(..., 19, 2)` (start well past len) → **undef** (with a warning). The PLAN's clamp `seq[min(p,n)..min(p+len,n)]` returns `""` in BOTH cases. This is a behavioral divergence (undef vs empty string) — **but I verified it is byte-identity-neutral**: `process_sequence` on undef runs `foreach (0..(length undef)-1)` = `0..-1` = empty loop (length(undef)=0 under `no warnings`), contributing zero counts, exactly like the empty-string case. So the PLAN's clamp produces identical counts. Worth a one-line note in C3 confirming "undef-vs-empty is observably identical because `process_sequence` no-ops on both." Currently SPEC §7.4 lumps them as "empty" without the undef distinction — accurate in outcome, imprecise in mechanism.

4. **(Important) SE/PE detection divergence is real but low-risk.** SPEC §7.1 / PLAN C5 use `bismark_io::detect_paired_from_header` which matches the first `@PG` with **`ID:Bismark`** (`read.rs:665`), whereas Perl `test_file` (`:259-271`) matches the **first `@PG` regardless of ID** and checks for `\s-1\s+` AND `\s+-2\s`. Divergence scenarios: (a) a BAM whose first `@PG` is e.g. `samtools` (no `-1`/`-2`) followed by a Bismark `@PG` — Perl returns SE (first @PG has no -1/-2), Rust skips non-Bismark @PG and reads the Bismark one → could return PE. (b) No `@PG` with `ID:Bismark` at all → Rust returns `None` → error; Perl, if there's ANY @PG, returns SE (never errors as long as one @PG exists), and only dies if zero @PG. For real single-Bismark-@PG BAMs these are equivalent (Q5 resolves "equivalent"). The risk is a re-processed BAM (e.g. piped through samtools sort, which prepends a `samtools` @PG line). **Real-world risk: moderate** — `samtools sort`/`view` DO add @PG lines, and a sorted-then-bam2nuc'd file would have `samtools` as the first @PG. Recommend: F1 (oxy) should include at least one BAM that has been through `samtools sort`/`view` (or document that the gate BAMs are raw Bismark output only). Note also: bam2nuc explicitly handles coordinate-sorted input (`without_sort_check`), so sorted-then-re-headered BAMs are a plausible input.

5. **(Important) The PE `or 163` revcomp applies to a span that may be empty.** Edge interaction: a PE read on a missing chr → `extract_span` returns `Vec::new()`; `correct_pe` revcomps an empty vec → empty; `process_sequence` no-ops. Fine. But ensure the C5 test "read on missing chr contributes nothing" uses a PE flag ≠ 99/147 (e.g. 83) to also exercise the revcomp-of-empty path. Currently C5 lists "read on missing chr contributes nothing" without specifying SE/PE.

6. **(Minor) `--genomic_composition_only` cache-write vs reuse ordering (Task E1 / B4).** Perl `get_genomic_frequencies` checks existence ONLY against `genome_folder` (`:163`), and on the genomic-composition-only path the same function runs (`:52-53`). So if a cache already exists in `genome_folder`, `--genomic_composition_only` **reads it and re-writes nothing** (the `if -e` branch has no write). The PLAN's E1 says "computes+writes (or reuses)" which is correct, but the B4 GREEN pseudocode for the reuse branch only "read lines… return" — it does NOT re-write. Good, that matches Perl. Just confirm the E2 golden cell 1 (`--genomic_composition_only`) is run against an EMPTY genome_folder (no pre-existing cache) so it exercises the compute+write path, and cell 5 (cache-reuse) exercises the read path. The PLAN implies this but doesn't state the genome_folder must start clean for cell 1.

7. **(Minor) `total_number_of_words_counted` global is dead in Perl.** Perl declares `my $total_number_of_words_counted = 0;` at file scope (`:46`) but `calculate_averages` re-declares a lexical `my $total_number_of_words_counted;` (`:280`) shadowing it. No effect on output. The PLAN correctly ignores it (separate mono/di totals are local). No action.

## Assumptions

- **Stated & validated:** counting is commutative (chr/read order irrelevant) → HashMap is fine; `seqs()` accessor order-agnostic (PLAN A3 correctly drops c2c's "no-iterator" invariant — that invariant existed only to protect c2c's covered-vs-uncovered output ordering, which bam2nuc has no analogue of). ✓ Sound.
- **Stated & validated:** the cache-reuse precedence makes the genomic column identical between a Perl-then-Rust comparison (SPEC §10) — true, and it is the linchpin of byte-identity for the genomic column. But see Important #4 below on the gate's cache handling (OI-3).
- **Implicit, needs surfacing:** the PLAN assumes `noodles_bam::io::Reader` over a plain `File` (single-threaded BGZF). At 55M reads the BAM decode is the dominant cost. The extractor went to lengths (#884) to add a 2-thread BGZF decoder for exactly this. bam2nuc's per-read work is far lighter than the extractor's (no XM parsing, no M-bias), so single-threaded decode may be acceptable, but the PLAN should state the throughput assumption explicitly (it's a QC tool, not on the hot path) rather than leave it implicit. Not a byte-identity concern.
- **Implicit:** the PLAN assumes the raw `RecordBuf` path will NOT silently drop unmapped reads (unlike `bismark-io`'s `filter_unmapped_then_classify`, `read.rs:580-594`, which drops FLAG&0x4). I confirmed `record_bufs` does NOT filter — raw iteration yields every record including unmapped. Good: this preserves the SE die-on-stray-flag contract. But it also means an unmapped SE read (flag 4) reaches `correct_se` → `Err(InvalidSeFlag{4})` → the whole run errors. Perl behaves identically (flag 4 ∉ {0,16} → die). **Faithful, but a real Bismark BAM has no unmapped reads, so this only bites malformed input** — which is the intended contract. Confirm with the C5 stray-flag fixture (the PLAN flags this in OI-1 / line 247 — good).
- **Implicit:** `sequence().len()` (`RecordBuf::sequence`) equals Perl's `length$sequence` (SAM column 10 length). For a record with `*` SEQ, noodles yields a zero-length sequence → span length 0 → no counts. Perl: column 10 = `*`, `length "*"` = 1 → `substr($chr, start-1, 1)` → ONE base counted! **Divergence.** A read with `*` SEQ but a non-I/D/S/N cigar (e.g. a `5M` cigar with `*` seq — unusual but constructible) would count 1 genomic base in Perl, 0 in Rust. Real Bismark BAMs always have a real SEQ, so unreachable on real data, but it is a genuine byte divergence on adversarial input that the SPEC/PLAN do not mention. Minor — document in D-deviations if you want completeness; otherwise note it's unreachable.

## Efficiency analysis

- **Genome in RAM (~3GB):** `HashMap<Vec<u8>, Vec<u8>>` of uppercased sequences — identical to c2c and Perl `%chromosomes`. Fine. The uppercase-on-load copy is one pass. ✓
- **`compute_genomic` (B3):** one `process_sequence` pass over every chromosome = O(genome length) with 2 HashMap bumps per base (1 mono + 1 di). For 3GB that's ~6 billion HashMap operations. **This is the expensive part** (Perl warns "may take several minutes"). The cache makes it a one-time cost. The PLAN's `NucCounts = HashMap<Vec<u8>, u64>` with a `Vec<u8>` key allocates a 1- or 2-byte `Vec` **per bump** if `bump(&[u8])` does `entry(key.to_vec())`. At 6 billion bumps that is 6 billion tiny allocations even with mimalloc. **Important efficiency note:** the genomic composition only has ≤ a few hundred distinct keys (4 mono + 16 di for ACGT, plus a handful of IUPAC). The hot path should look up by `&[u8]` WITHOUT allocating a `Vec` on every hit (e.g. `if let Some(v) = map.get_mut(key) { *v += 1 } else { map.insert(key.to_vec(), 1) }`, or better, a fixed `[u64; 256]` for mono + `[[u64;256];256]` or a `HashMap` keyed on a `u16` packed di). The PLAN's B1 GREEN (`counts.bump(&seq[i..i+1])`) is correct but the `bump` implementation must avoid per-call allocation or the genomic pass will be needlessly slow. **This won't affect byte-identity, only speed** — but it's the one place the QC tool could be embarrassingly slow vs the Perl it replaces if implemented naively. Flag for B1/B2 GREEN: specify the allocation-free bump.
- **Per-read loop (C5):** 55M reads × (span extract = 1 slice copy of ~100 bytes + ~200 HashMap bumps). The span `.to_vec()` (C3) is a per-read alloc; revcomp another. Acceptable for a QC tool. mimalloc (A1) helps. The sample `NucCounts` also has ≤ a few hundred keys → same allocation-free-bump concern applies but the volume (55M×200 = 11B bumps) is comparable to the genomic pass. Worth the same allocation-free treatment.
- **`to_cache_string` (B2):** collect ≤ few-hundred keys, sort, format. Trivial. ✓

**Recommendation:** make `NucCounts::bump` allocation-free on the hot path (lookup-then-maybe-insert, or a packed-byte array). This is the single highest-leverage efficiency item and is currently glossed in B1/B2.

## Validation sufficiency

The local-golden + oxy-gate two-tier strategy is the right shape (mirrors c2c/bedgraph) and the 7 local golden cells (E2) plus 4 oxy cells (F1) cover the main risks. Where wrong output could still slip through:

1. **(Critical) The empty-count-field is only tested locally, never on the oxy gate.** OI-2 correctly flags that the empty-field (`AA\t\t0.00\t…`) only bites tiny samples — on real data every word appears. So the **oxy gate cannot catch a regression in the empty-field logic** (real data fills every cell). The local E2 SE golden MUST contain a sample missing ≥1 di-word, AND D2's unit test must assert the literal empty field. The PLAN has both (D2 RED "missing sample di-word → empty field" + E2 implies it) — but make it a **hard requirement**, not "the SE golden will prove it." I verified Perl: an absent `%freqs{$word}` interpolates to `""` in the count column and coerces to 0 in the math (`pct=0.00`). The most likely implementer error is printing `0` instead of `""`. Pin a golden line `\bAA\t\t` (tab-tab) explicitly.

2. **(Important) Division-by-zero is asserted as an error but its trigger on real data is impossible** → the oxy gate won't exercise it. D2's unit tests (zero mono total, zero genomic count) are the only coverage. I confirmed Perl dies "Illegal division by zero" on both. Q6 resolves to hard-error. The unit tests are sufficient since this is a pure-function property. ✓ But note: the EMPTY-sample case (all reads skipped/InDel) producing a zero mono total is a plausible real edge (a BAM where every read has an InDel) — worth one local golden cell where the sample is entirely InDel-skipped → expect the Rust binary to exit 1 (and Perl to die). The PLAN's E2 has an InDel-skip cell but it's a partial skip; add/confirm an all-skipped cell to exercise the ZeroDivision exit-code path end-to-end, not just as a unit test.

3. **(Important) No golden cell pins the genomic-vs-sample column relationship under cache-reuse with DIFFERING planted values.** B4 RED does this at the unit level (plant a cache whose counts differ from the genome, assert planted wins). E2 cell 5 plants a cache and runs SE. Good — but ensure cell 5's planted cache has values that **could not** arise from the tiny genome (e.g. inflate by ×1000), so a bug that recomputes-instead-of-reusing would visibly fail the byte compare. The PLAN says "assert stats use planted genomic column" — make the planted values obviously synthetic.

4. **(Important) IUPAC-in-genome cache golden (E2 cell 2) is the ONLY guard for the count-everything-not-just-ACGT rule.** This is the rule most likely to be "optimized" wrong (an implementer adding an ACGT filter to `process_sequence` would pass every test EXCEPT this one and the stats would still look right because stats only read ACGT rows). The cell is necessary and well-chosen. Ensure the fixture genome has an IUPAC base positioned so it produces both a mono IUPAC row AND di IUPAC rows (e.g. `...ACRGT...` → mono `R`, di `CR` and `RG`). The PLAN says "+ one IUPAC base in a dedicated cell" — confirm it generates ≥1 di IUPAC row, not just the mono.

5. **(Moderate) Platform rounding (F1) — adequate.** My macOS test showed Rust==Perl on all ties. The residual is Linux libc. F1 runs Perl AND Rust on oxy and diffs → if Linux libc rounded differently, BOTH Perl-on-oxy and the local-Perl-goldens would diverge from each other, surfacing it. Actually the stronger guarantee: F1 compares Rust-on-oxy vs Perl-on-oxy (same platform), so any platform rounding shift affects both identically → byte-identity holds regardless. The ONLY way rounding breaks the gate is if Rust core and oxy-libc disagree on a tie — possible in principle but I found no such case. **Adequate.** One enhancement: F1 should pick at least one real BAM whose A/C/G/T percentages land near a half-way tie (hard to engineer on real data, so this is best-effort) — or accept that the unit-test ties (D2) plus same-platform diff (F1) jointly cover it.

6. **(Minor) Multi-input loop (`@ARGV` / Task 37) coverage.** E1 has a "two .bam inputs → two stats files; cache written once, reused for the second" cell. This is the only test of the per-file `%freqs` reset (SPEC §4) and the cache-reuse-after-file-1. Good. Ensure the two inputs produce **different** sample counts so a `%freqs`-not-reset bug (carrying file 1's counts into file 2) would fail. The PLAN doesn't say the two BAMs differ — make them differ.

7. **Gap: no test for a BAM with a non-Bismark first `@PG`** (the SE/PE detection divergence, Important #4 above). Add either a unit test on `detect_paired_from_header` with a `samtools`-then-`Bismark` header, or document the divergence as accepted.

## Alternatives

1. **Reader path (OI-1).** The PLAN's resolved choice (raw `noodles_bam::io::Reader::record_bufs`) is correct and I confirmed the API. The alternative — routing through `bismark-io::BamReader` — is rightly rejected: it requires XR/XG/XM (`record.rs:116-130`) which bam2nuc doesn't use, AND it silently drops unmapped reads (`read.rs:586-590`), which would mask the SE stray-flag die. **One sub-alternative the PLAN should weigh:** `bismark-io` could grow a thin `raw_record_bufs()` passthrough on `BamReader` (returning `RecordBuf` un-classified). That would keep bam2nuc free of a direct `noodles-bam` dep and centralize BGZF/threading. But it adds API surface to a shared crate for one consumer. Given the Critical #1 (the dep is missing anyway), I lean toward the PLAN's local-reader approach but the implementer should at least note the bismark-io-passthrough option in the commit. Either way, **add `noodles-bam` + `noodles-bgzf` to deps** (Critical #1).

2. **`NucCounts` representation.** PLAN uses `HashMap<Vec<u8>, u64>`. Alternative: `[u64; 256]` for mono + `[u64; 65536]` (di packed as `(b0 as u16)<<8 | b1`) for di — fully allocation-free, O(1), and the cache serialization just iterates the populated cells in byte order (which is naturally sorted). This would eliminate the entire allocation-on-bump concern (Efficiency above) AND make the bytewise sort free. The HashMap is simpler and matches Perl's mental model; the array is faster and removes a footgun. **Recommend the array for `bump`** but it's an optimization, not a correctness requirement. (Caveat: the di array is 512KB — fine. Mono+di N-skip logic is unchanged.)

3. **CRAM (Q3, deferred).** Correctly deferred. The `AlignmentKind::Cram` reject in E1 is the right v1.0 stance. No alternative needed.

4. **Genomic-composition-only could skip the genome HashMap entirely** and stream-count each FASTA file (Perl builds the full `%chromosomes` first, then iterates). The PLAN follows Perl (load all, then iterate `seqs()`). Faithful and fine; streaming would save the 3GB residency but is a non-trivial divergence from the shared genome reader. Keep as-is.
