## 1. Logic review

### 1.1 Byte-identity-critical claims — verified line-by-line

**PE flag "latent bug" (Task C4 / SPEC §7.5 / D3) — CORRECT.** Perl `:226` `elsif ($flag == 83 or 163)`. I ran this live: because `==` binds tighter than `or`, this parses as `(($flag == 83) or 163)`, and `163` is a truthy constant, so the `elsif` is **always true**. Confirmed for flags 83, 163, 256, 0, 1024 — every one hits the revcomp branch; the `else{die}` (`:231-233`) is dead code. The PLAN's `correct_pe`: `if flag == 99 || flag == 147 { span } else { revcomp(&span) }` replicates this exactly, never erroring. ✅

**SE flag logic (Task C4) — CORRECT.** Perl `calc_single_end` `:240-250`: flag 0 → as-is, flag 16 → revcomp, **else → reachable `die`**. The `else{die}` here is genuinely reachable (no truthiness trick — both arms are real `==` comparisons). PLAN's `correct_se` returning `Err(InvalidSeFlag{flag})` for anything other than 0/16 is faithful. ✅

**reverse-complement (Task C4) — CORRECT.** Perl `:228-229` / `:245-246`: `reverse $sequence` then `tr/GATC/CTAG/`. The translation map is G→C, A→T, T→A, C→G; N and every other byte untouched (Perl `tr` leaves un-listed chars alone). PLAN's `revcomp` match arm `{ b'G'=>b'C', b'A'=>b'T', b'T'=>b'A', b'C'=>b'G', x=>x }` over `.rev()` is exact. The PLAN's test vectors check out: `b"GATC"` → rev `CTAG` → tr → `GATC` ✅; `b"AACG"` → rev `GCAA` → tr → `CGTT` ✅. **Note** the operation is uppercase-only by design (genome is uppercased on load, §5), so lowercase g/a/t/c never reach `revcomp` — fine, but see Important I-4.

**Genomic span extraction (Task C3) — MOSTLY CORRECT, ONE OFF-BY-ONE GAP.** Perl `:133` `substr($chromosomes{$chr}, $start-1, length$sequence)`. I verified Perl's saturation live against `chr="ACGTACGT"` (len 8):
- POS=1,len=4 → `"ACGT"` ✅ (PLAN matches)
- POS=6,len=4 → `"CGT"` (3 bytes, truncated) ✅ (PLAN matches)
- POS=9,len=2 (start-1 = 8 == len) → `""` ✅ (PLAN matches: `start=min(8,8)=8, end=min(10,8)=8 → seq[8..8]=""`)
- **POS=20,len=2 (start-1 = 19 > len) → Perl returns `undef`** (with an "uninitialized value" warning), NOT `""`.

  The PLAN's clamp gives `start=min(19,8)=8, end=min(21,8)=8 → seq[8..8] = ""`. So the PLAN produces empty `""` where Perl produces `undef`. **Is this byte-divergent?** I checked downstream: `undef` flows into `calc_single_end`/`calc_paired_end` → `process_sequence`, where `length$seq` of `undef` is 0 (the `foreach my $index (0..-1)` loop runs zero times). So `undef` and `""` contribute **identically zero counts**. ⇒ The off-by-one is **behaviourally equivalent for counting** and does NOT break byte-identity. **BUT** the PLAN's Task C3 RED tests only cover POS=9 (start == end) and chr-absent; they do **not** include a "start strictly past end" cell (POS > len+1). Without that test, a future refactor that changed the clamp (e.g. to `start = p` un-clamped, panicking) wouldn't be caught. → Important I-1: add a POS-far-past-end test cell asserting empty output.

**CIGAR `[IDSN]` skip (Task C2) — CORRECT.** Perl `:126` `if ($cigar =~ /[IDSN]/)`. The regex matches any I/D/S/N letter anywhere in the CIGAR string. The PLAN's structured equivalent {Insertion, Deletion, SoftClip, Skip} is exact, and noodles op `Kind` has exactly these variants (verified in `bismark-io/src/cigar.rs:159-203`: `Kind::Insertion`, `Kind::Deletion`, `Kind::SoftClip`, `Kind::Skip`, plus `HardClip`/`Pad`/`Match`/`SequenceMatch`/`SequenceMismatch`). H/P/=/X are correctly NOT skipped (Perl regex `[IDSN]` doesn't match H/P/=/X). PLAN test vectors (`5M2H`→keep, `5M2N5M`→skip, etc.) are all correct. One subtlety: Perl's regex would also match the literal letters if they appeared in a bizarre CIGAR; noodles parses structurally, so a malformed CIGAR string that Perl's regex would catch differently cannot arise from a valid BAM. Benign. ✅

**`process_sequence` counter (Task B1) — CORRECT.** Perl `:318-343`:
- Mono `:330-333`: `$mono = substr($seq,$index,1); unless ($mono eq 'N'){ $freqs{$mono}++ }` — counts everything except literal `N`. IUPAC codes (R/Y/etc.) ARE counted. ✅
- Di `:335-340`: `unless (($index+2) > length$seq)` guards the window; `$di = substr($seq,$index,2); if (index($di,'N') < 0){ $freqs{$di}++ }` — skips any 2-mer containing N, counts everything else. Overlapping windows, L-1 of them. ✅
- PLAN's loop `for i in 0..seq.len() { if seq[i]!=b'N' {bump mono}; if i+2<=len { if !di.contains(N) {bump di} } }` is exact. The "counts IUPAC" requirement is correctly carried into the cache (cache must include R/AR/RA rows) per Task B2/B3. The PLAN test `"ARG"` → mono A,R,G; di AR,RG is correct. ✅

**Cache file format/sort (Task B2/B3/B4) — CORRECT.** Perl `:191` `foreach my $f(sort keys %genomic_freqs)`. Perl's default `sort` is bytewise (string `cmp`) under `LC_ALL=C` (the PLAN pins `LC_ALL=C` in the golden script — essential, see I-2). The PLAN's `sort_unstable()` on `Vec<u8>` keys is bytewise and matches. The 20-line ACGTN ordering (`A, AA, AC, AG, AT, C, CA, …, T, TA, TC, TG, TT`) is correct: a 1-char prefix sorts before its 2-char extensions because byte comparison reaches end-of-string first. IUPAC keys (`R`, `AR`) sort into position correctly (`A < AR` since `AA..AT < AR`? — actually `R`(0x52) > `G`(0x47) so `AR` sorts after `AG`/`AT` and before `C`; `R` mono sorts after `G`+its di but before... — the exact interleaving is automatic from bytewise sort and the PLAN delegates to `sort_unstable`, so it's correct by construction). ✅

**Cache precedence (Task B3/B4) — CORRECT.** Perl `:163` checks existence ONLY at `${genome_folder}genomic_nucleotide_frequencies.txt`; on miss it computes then tries genome_folder (`:189`), else output_dir (`:203`), else warn-and-continue (`:212`). The "existence check only against genome_folder" quirk (so a cache that landed in output_dir triggers recompute next file) is faithfully captured in Task B4. ✅

**Stats output (Task D2) — CORRECT incl. the empty-count-field.** I verified the empty-field claim live: Perl `:295`/`:296` interpolates `$freqs{$word}` directly into the output string. For a word never incremented, `$freqs{$word}` is `undef`, which interpolates as the **empty string** in the count column (producing `AA\t\t0.00\t…`), while `:292` `100*$freqs{$word}/$total` coerces undef→0 (giving `0.00`). My live run produced `>AA\t\t0.00\n<` — exactly the PLAN's claimed `AA\t\t0.00\t<g>\t<gp>\t0.000`. The PLAN's `cs.map(|n|n.to_string()).unwrap_or_default()` for the printed field + `cs.unwrap_or(0)` for the math is exact. ✅ Header text `:284`, mono order A,C,G,T `:286`, di order `:302`, separate mono/di totals (`:287-289` reset to 0 at `:299-300` then re-sum `:303-305`), `%.2f` pct, `%.3f` coverage — all match the PLAN. ✅

**Division-by-zero (Q6/§8.3) — CORRECT.** Perl `:292`/`:294` `100*$freqs{$word}/$total` and `$freqs{$word}/$genomic_freqs{$word}` both `die "Illegal division by zero"` when the denominator is 0 (empty/all-skipped sample, or a genome word with count 0). PLAN's `ZeroDivision` hard error matches. Note: Perl dies AFTER writing the header line + possibly some rows to OUT (it dies mid-`calculate_averages`), leaving a partial file. The PLAN should document that the Rust port likewise need not clean up a partial stats file (parity with c2c's documented "partial file left on die" behaviour, c2c error.rs:132-141). → Optional O-1.

**`%.2f`/`%.3f` rounding (§8.3/§10) — CLAIM HOLDS on macOS; oxy gate still needed.** The PLAN asserts Rust `format!("{:.2}")` == round-half-to-even == C `printf`. I ran a live macOS comparison (Rust 1.x `rustc -O` vs system Perl `sprintf`) across `0.125, 0.135, 0.145, 2.5, 0.5, 1.005, 30.905, 30.915, 0.0045, 0.0035` AND realistic ratios `100*14541/47050` etc. — **every value byte-identical** between Rust and Perl. Crucially `30.905 → .2 = 30.91` in BOTH, confirming both round on the true f64 bits (not naive decimal), which is the parity that matters. So the local D2 parity test will pin macOS. The SPEC's caution that "the dev platform cannot adjudicate a platform-specific contract" is still valid (glibc vs macOS libm tie-breaking could in principle differ on a pathological value), so F1 on oxy remains the real gate. Risk: **Low.** The claim is sound; F1 closes it.

### 1.2 Read-path logic (Task C5 / OI-1) — ARCHITECTURE SOUND, ONE COMPILE-BLOCKING DEP GAP

The raw-`RecordBuf` decision is the right call and is **achievable**. I verified:
- `bismark-io` does NOT re-export `noodles_bam` (lib.rs:17-39 — no `pub use noodles_bam`), and its only BAM record API (`BamReader::records()`, read.rs:257-262) routes through `BismarkRecord::from_noodles_record`, which **requires XR/XG/XM** (record.rs:116-130) and whose iterator layer **silently drops unmapped reads** (read.rs:580-594). So bismark-io cannot supply the tag-agnostic, unmapped-preserving path bam2nuc needs.
- Therefore the crate MUST depend **directly** on `noodles-bam` (and `noodles-bgzf` at runtime) and call `noodles_bam::io::Reader::new(BufReader<File>)` → `read_header()` → `record_bufs(&header)`. I confirmed these exist in noodles-bam 0.89.0 (`reader.rs:187 read_header`, `:287 record_bufs<'r,'h>`). Raw `record_bufs` does **not** filter unmapped and does **not** sort-check — so the SE "die on stray flag" contract is preserved (an unmapped SE read has flag 4 → `correct_se` returns `Err(InvalidSeFlag{4})`, faithful to Perl's `die`). ✅ This resolves OI-1: the cleanest path is **direct noodles-bam**, NOT a bismark-io passthrough (none exists).
- **GAP (Critical C-1):** Task A1's Cargo.toml dep list is `bismark-io, clap, thiserror, noodles-fasta, flate2, noodles-sam, mimalloc` + dev-deps `…noodles-bgzf…`. It lists **neither `noodles-bam` as a regular dep nor `noodles-bgzf` as a regular dep**. The chosen read path cannot compile without both as **runtime** deps. Pin `noodles-bam = "=0.89.0"` (the workspace lock version; bismark-io already uses it) and `noodles-bgzf = "=0.47.0"` (currently dev-only). Also: since the crate constructs `noodles_sam::Header` objects (for the chr-name table and tests) AND passes a header into `noodles_bam::record_bufs`, both must agree on the noodles-sam version. The lock has a single `noodles-sam 0.85.0`; noodles-bam 0.89.0 is compatible with it in the current lock (verified: one noodles-sam entry). Keep noodles-sam pinned `=0.85.0` to avoid pulling a second copy. **Without C-1 fixed, Phase C does not build.**

### 1.3 SE/PE detection (Task C5 / Q5) — divergence is real but benign

PLAN uses `bismark_io::detect_paired_from_header` (read.rs:649-677), which matches the **first `@PG` line containing `ID:Bismark`** and checks for `-1`/`--1` AND `-2`/`--2` as space/tab-delimited tokens. Perl `test_file` (`:255-273`) matches the **first `@PG` regardless of ID** and checks `/\s-1\s+/` AND `/\s+-2\s/`. Two divergences:
1. **First-`@PG` vs first-Bismark-`@PG`:** if a non-Bismark `@PG` (e.g. `samtools`) precedes the Bismark one, Perl inspects the samtools line (finds no `-1`/`-2` → declares SE), while the Rust helper skips to the Bismark line. On a real Bismark BAM the Bismark `@PG` is first, so identical. **But** a coordinate-sorted-then-`samtools-sort`-stamped BAM, or any post-processed BAM with a leading non-Bismark `@PG`, would diverge: Perl→SE, Rust→PE. This is an **accepted, documented** divergence and arguably the Rust behaviour is *more* correct. Confirm it's listed in §12 (it is, as Q5/the §7.1 note). The real-data gate (F1) on actual Bismark BAMs won't expose it. → Optional O-2: add a unit fixture with a leading `samtools` `@PG` + trailing `Bismark` PE `@PG`, asserting Rust says PE, and note the Perl divergence in a comment so it's not "discovered" later as a bug.
2. **Token boundary:** Perl `\s-1\s+` requires whitespace before AND after; the Rust `arg_present` requires a delimiter on both sides too (read.rs:687-696, with the documented quirk that `-1` at absolute end-of-line is "not present"). Equivalent for Bismark's real CL (always `-1 <path>`). ✅

### 1.4 Output filename (Task D1 / §9 / D4) — CORRECT

Perl `:148` `$outfile =~ s/.*\///` (basename), then `:150` `s/(bam|cram)$/nucleotide_stats.txt/` with **no dot anchor**. PLAN replicates: strip trailing token `bam`/`cram` (not `.bam`), append `nucleotide_stats.txt`, else `Err(NotBamOrCram)`. The no-dot quirk test `foosubbam → foosubnucleotide_stats.txt` is correct. ✅ One subtlety the PLAN handles well: `.sam` makes the substitution fail → Perl dies; the Rust gate (E1) rejects `.sam` earlier. The PLAN's note about the benign reordering (reject before counting vs Perl deriving the name after counting) is sound — a `.bam` always names cleanly so the observable outcome is identical.

### 1.5 Top-level flow (Task E1 / §4) — CORRECT, one ordering note

`run()` matches Perl `:48-61`: load genome, `--genomic_composition_only` → compute/reuse cache then exit, else per-file loop (reset sample counts, reuse cache after file #1, count, write stats). The `%freqs` reset-per-file is captured (`get_genomic_frequencies` returning a fresh `NucCounts` per file). ✅

**Empty-genome edge case (not in PLAN):** the c2c genome reader returns a NON-empty-error empty `Genome` when the only FASTA is `Mus_musculus.NCBIM37.fa` (genome.rs:272-277 `mus_only_tier_yields_empty_genome_no_error`). In `--genomic_composition_only` mode over an empty genome, Perl writes a **0-byte** `genomic_nucleotide_frequencies.txt` (I confirmed: `sort keys %genomic_freqs` over an empty hash → 0 lines). The PLAN's `compute_genomic` over `genome.seqs()` (empty iterator) → empty `NucCounts` → `to_cache_string()` → `""` → 0-byte file. So this is byte-correct **by construction**, but it's untested. → Optional O-3: an empty-genome (Mus-only dir) `--genomic_composition_only` cell asserting a 0-byte cache file. Low priority (cannot occur on a real genome dir).

### 1.6 `seqs()` accessor (Task A3) — sound, but note the c2c invariant being dropped

The c2c `genome.rs` has an explicit module-level invariant (lines 15-20): "exposes NO public insertion-order iterator … Do NOT add an `iter()`/`keys()` passthrough." That invariant exists to protect c2c's byte-identity (covered-chromosome OUTPUT order). The PLAN correctly reasons that bam2nuc's genomic counting is **commutative** (di-windows never cross chr boundaries, §6.3), so order is irrelevant and a `seqs()` accessor is safe HERE. ✅ This is correct, but since the PLAN "copies c2c genome.rs", the implementer must **delete the c2c invariant comment** when adding `seqs()` (otherwise the code contradicts its own docs). → Optional O-4: call this out in Task A3 so the copied comment isn't left stale.

## 2. Assumptions

**Validated:**
- A-1 (PLAN): "Perl bam2nuc runs on macOS" — true (it's pure Perl + shells out to `samtools view`; the local oracle uses `/opt/homebrew/bin/samtools`). The golden script's dependency on a local `samtools` is real but satisfied. ✅
- A-2: "counting is commutative ⇒ chr/read order irrelevant" — true; di-windows are within-chromosome and within-read. ✅
- A-3: "Rust `{:.2}` == C printf" — validated on macOS (§1.1); oxy confirms cross-platform. ✅
- A-4: genome uppercased on load ⇒ revcomp only sees uppercase ACGTN+IUPAC. ✅ (c2c genome.rs:201 `to_ascii_uppercase`.)

**Implicit / under-stated assumptions to surface:**
- **AS-1 (Important I-3):** The PLAN assumes `noodles_bam::record_bufs` yields the SAME field values Perl extracts from `samtools view -h` text. One risk area: **`sequence().len()`**. Perl uses `length$sequence` where `$sequence` is column 10 (SEQ). For a read with `SEQ == "*"` (no sequence stored), Perl's `length("*")` = 1, while noodles `RecordBuf::sequence().len()` = 0 for `*`. A `*`-SEQ read with a linear CIGAR (no I/D/S/N) is pathological (Bismark never emits it), but if it occurred, Perl would extract a 1-byte span and Rust a 0-byte span → divergence. Bismark BAMs always have a real SEQ, so this is theoretical, but the PLAN should state the assumption "SEQ is always present and CIGAR-consistent on Bismark BAMs" explicitly. → I-3.
- **AS-2:** The PLAN assumes `reference_sequence_id → chr name` via the `@SQ` order equals the chr names Perl reads from column 3. Perl uses the **literal chr string** from the SAM line (col 3); the Rust path resolves the numeric `reference_sequence_id` through the header `@SQ` table (Task C1, mirroring extractor header.rs). These agree iff the BAM's `@SQ` names match the genome FASTA names — which is the standard Bismark invariant. A read whose `reference_sequence_id` is `-1` (unmapped, `*`) would index out of the table; the PLAN must guard this (an unmapped read has chr `*` → `genome.get(b"*")` → None → empty span; but the numeric id is `-1`/`None`). The PLAN's `reference_sequence_id → chr name` step needs an explicit "None id → treat as absent chr → empty span" branch to mirror Perl's `substr($chromosomes{"*"}, …)` → undef. → folds into I-3.
- **AS-3:** `--samtools_path` no-op (Q4). Perl actually *validates* the samtools path exists (`:436-453` `die "Could not find an installation of Samtools"`). The Rust port accepts-and-ignores, so a user passing `--samtools_path /bad/path` gets NO error in Rust but a `die` in Perl. Since neither emits this to the gated output files, it's byte-neutral. Correctly out of scope, but the divergence (Rust won't validate the path) should be in §12/D1. → Optional O-5.

## 3. Efficiency analysis

- **Genome in RAM (~3 GB):** `Genome` stores `HashMap<Vec<u8>, Vec<u8>>` of uppercased sequences — same as c2c, proven at full-hg38 scale on oxy (per the c2c memory note). Acceptable. mimalloc (Q8) is byte-neutral and helps the per-read allocation churn.
- **Per-read hot loop (~55M reads):** The dominant cost is `extract_span` (`seq[start..end].to_vec()` — one allocation per read) + `process_sequence` (HashMap bump per mono + per di). **Concern E-1 (Important):** `NucCounts` as `HashMap<Vec<u8>, u64>` allocates a `Vec<u8>` **key on every `bump`** for the lookup (the PLAN's `bump(&[u8])` takes a slice, but `HashMap<Vec<u8>,_>::entry` requires an owned `Vec<u8>` key, forcing a per-call allocation unless using the `raw_entry`/`get_mut`-then-`insert` dance). At ~55M reads × ~100 bp × 2 (mono+di) ≈ 11 billion `bump` calls, a per-call key allocation would dominate runtime. The Perl uses a hash too, but the Rust port's whole value proposition is speed. **Recommendation:** the key space is TINY and bounded (≤ ~25 distinct words for ACGTN, a few more with IUPAC). Use a fixed small structure — e.g. a `[u64; N]` indexed by a perfect-hash of the 1-or-2 nucleotide bytes, or `get_mut` first and only allocate a key on the (rare) first-insert. The PLAN's `freqs.rs` design should specify the non-allocating bump explicitly; otherwise the "6-10x faster than Perl" goal is at risk. This does NOT affect byte-identity (counts are identical), so it's an efficiency Important, not a Critical. → E-1.
- **Genomic composition pass:** `compute_genomic` runs `process_sequence` over every chromosome (≈3 Gbp). Same `bump` concern × genome size. With the cache-reuse precedence, this runs **once** (first file writes the cache; subsequent files reuse). On the gate, if Perl wrote the cache first, Rust reuses it — so the expensive genome pass may not even run in the comparison. Still, the first-ever run pays it; the non-allocating `bump` (E-1) matters here too.
- **No parallelism proposed** — correct for a v1.0 byte-identity port (bam2nuc is ~16% +9% of pipeline per CLAUDE.md profiling; the per-read loop is I/O-light since it reads BAM once). Threaded BGZF decode (à la extractor #884) is a reasonable future optimization but out of scope. Noting for completeness, not a finding.

## 4. Validation sufficiency

**Strong coverage.** The local Perl-oracle golden harness (E2) with 7 cells + the oxy real-data gate (F1) hit the highest-risk divergences. Gaps where wrong output could slip through untested:

- **V-1 (Critical-adjacent → Important):** No golden cell exercises the **PE `or 163` latent bug with a flag that is neither 99/147/83/163** (e.g. a PE flag 0, 137, or 1024). The unit test C4 covers flag 0 (PE)→revcomp and 256→revcomp, but the **golden** (full Perl-oracle) cells (E2 cell 4) only use the canonical 99/147/83/163. The bug's entire point is that *unusual* flags also revcomp. A hand-built PE BAM with a supplementary/secondary flag (e.g. 2147=0x83+secondary) run through BOTH Perl and Rust would be the definitive proof. → V-1: add a PE golden cell with a non-canonical flag.
- **V-2 (Important):** No test pins the **"start strictly past end → empty"** substr case (see I-1). C3 tests start==end and chr-absent only.
- **V-3 (Important):** The **empty-count-field** behaviour (OI-2) is the single most fragile byte-detail (an extra/missing `\t` breaks the whole line). The PLAN flags it as OI-2 and plans a local SE golden with a missing di-word. Make this a **named, mandatory** golden cell, not an open item — the tiny SE BAM must be constructed so at least one of the 16 di-words is genuinely absent from the sample (easy: a 1-read SE BAM whose 4-bp span has only 3 di-words). Assert the exact bytes incl. the empty field. → V-3 (promote OI-2 to a hard requirement).
- **V-4 (Optional):** No test for **two input files where file #1's cache write fails** (read-only genome dir) so file #2 recomputes (the "existence check only against genome_folder" quirk, §6.1). B3/B4 unit tests cover precedence in isolation; an integration cell would prove the cross-file recompute. Low priority (byte-neutral since counts are deterministic).
- **V-5 (Optional):** `--samtools_path` no-op isn't tested (a CLI test passing a garbage `--samtools_path` and asserting success). Cheap to add given Q4. → folds into A2 tests.

The "RED→GREEN" framing per task is good; each task is independently testable. **Phase ordering is sound** (A scaffold → B counter/cache → C per-read → D report → E wiring → F gate): each phase's deps are satisfied by predecessors, and the shared `error.rs`/`lib.rs` growth justifies the single sequential stream (PLAN's parallel-stream note is correct — do NOT split).

## 5. Alternatives

- **ALT-1 (counts storage):** Instead of `HashMap<Vec<u8>, u64>`, use a flat `[u64; 21]` (5 mono + 16 di for ACGT) plus a small overflow `HashMap` for IUPAC words. This makes `bump` allocation-free and the hot loop branch-predictable, while still supporting the rare IUPAC cache rows. Trade-off: slightly more code; big win on the 11-billion-bump hot path. Recommended to fold into E-1's resolution.
- **ALT-2 (read path):** Use `bismark-io`'s `AlignmentKind::from_path` for the format **gate** (E1) — it's already the sibling-standard magic-byte sniff (read.rs:111-139) and correctly classifies `.sam`/`.cram`/`.bam` by content, not extension. The PLAN's E1 already proposes this (`match AlignmentKind::from_path(infile)?`). Good — keep it. But note `from_path` does a magic-byte sniff that OPENS the file; for the actual record reading you then re-open via `noodles_bam::io::Reader`. Two opens per file is fine (sibling precedent: read.rs:155 re-opens for the same reason). Just confirm the gate uses `from_path` (content) not `from_extension` — the PLAN says `from_path`. ✅ Minor: a `.bam`-content file with a `.sam` name would pass the gate (content-based) but then `derive_output_name` strips trailing `sam`??
