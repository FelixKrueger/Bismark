## Deviations / decisions during implementation (beyond SPEC §10)

1. **`allow_negative_numbers = true`** on the clap command — required IN ADDITION to signed
   `i64` types: signed types let the value *hold* `-1`, but clap's parser still rejects a
   leading-dash token as a flag unless this is set. Both together route `--threshold -1` to the
   Perl-style validation die (not a clap usage error). (Found via a failing unit test; fixed.)
2. **0-byte report on PE die** is realised by creating the report file empty up-front (mirroring
   Perl opening `REPORT` before the loop); the SUMMARY is written only on success.
3. **Valid partial BAMs on a mid-stream PE die**: both writers are `try_finish()`-ed regardless of
   the stream result before the error is propagated, so the prior complete pairs form a valid BAM
   (matching samtools finalising on pipe close). Verified by `edge_cases::pe_lone_trailing_r1_*`.
4. **SO-coordinate header reject** added before opening writers (faithful no-output for the common
   position-sorted PE case); the adjacent-qname check in the PE loop is the backstop (may leave
   partial output on the rare no-SO-but-misordered case — documented SPEC §10.5).
5. `--help` → clap exit 0 (SPEC §10.1 A1; **Felix-confirmed 2026-06-01: keep exit 0**); `--version` exit 0.
