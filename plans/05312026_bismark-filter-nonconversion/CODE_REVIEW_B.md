## Summary

This is a high-quality, faithful port. The byte-identity contract is well-defended: I
independently re-derived the load-bearing claims against live Perl 5.34 + samtools 1.21 and
they hold. Specifically I confirmed end-to-end against Perl:

- **Percentage rounding parity** across *every* `x.25`/`x.75` round-half-to-even tie up to
  denominator 64 (6.25→6.2, 18.75→18.8, 31.25→31.2, 43.75→43.8, …, 93.75→93.8), at 13
  straddling cutoffs — all byte-identical (`round_1dp` via `format!("{:.1}")` matches Perl
  `sprintf("%.1f")` exactly).
- **SE unmapped read** kept verbatim, in original order, body byte-identical + report identical.
- **Peek-then-chain** borrow pattern drops/duplicates **nothing** (5-record input → exactly
  5 kept, in order, record 1 not duplicated).
- The 9 committed goldens (generated by real Perl) pass the hermetic gate.

The XM decision, report formatting (SE 1-space vs PE 2-space, four Line-B variants, `N/A`,
`consecutive ` insert, timing arithmetic), filename derivation, CLI validation order, PE
lone-R1 die (0-byte report + valid partial BAMs), SO-coordinate reject, and the absent/empty-XM
PE die are all correct. Counter types are sound (per-read `u32` over an XM bounded by read
length; per-file `u64` totals — no realistic overflow).

I found **one genuine faithfulness divergence** (Medium, in a *shared* lib helper — likely
out of scope to fix here but must be on record) and a handful of Low items. **No Critical or
High issues. No bugs that affect real Bismark data on the gated paths.** I applied no source
changes (nothing met the unambiguous-low-risk bar without risking the byte contract or a
shared crate).

## Issues by area

### Logic / faithfulness

**B-1 (Medium) — Multiple Bismark `@PG` lines: Rust "first wins" vs Perl "last wins".**
Perl `determine_file_type` (lines 374–399) loops over **all** header lines and does **not**
`last` after finding a Bismark `@PG`; it overwrites `$single`/`$paired` each time, so the
**last** Bismark `@PG` line decides the mode. The shared helper
`bismark_io::detect_paired_from_header` (`rust/bismark-io/src/read.rs:649`) **returns on the
first** matching `@PG`. I reproduced a real divergence: a header with a PE-style
`@PG ID:Bismark` first and an SE-style `@PG ID:Bismark.1` last →

- Perl: prints "Treating … paired-end" then "Treating … single-end" → processes as **SE** (2 reads, success).
- Rust: detects **PE** on the first line → dies on the qname-mismatch backstop.

Severity is Medium, not High, because (a) it requires *two* Bismark-aligner `@PG` lines in one
header (unusual — Bismark aligns once; downstream tools use distinct IDs), and (b) it lives in
`bismark-io`, used identically by dedup/extractor, so a fix is cross-crate and out of this
crate's scope. **Recommendation:** do not patch `bismark-io` from this PR; instead either
(i) file a follow-up to make `detect_paired_from_header` honor last-match, or (ii) add a one-line
note to SPEC §4.4 / §10 documenting the divergence as accepted (matching how §10.7 documents
the empty-XM divergence). Also note the substring match `line.contains("ID:Bismark")` will
match `ID:Bismark.1`, `ID:Bismark2`, etc. — a separate (pre-existing, shared) looseness vs
Perl's identical `/ID:Bismark/`, so the two are actually equivalent *on the substring test* —
the only behavioural gap is first-vs-last.

**B-2 (Low) — Truncation detection only covers header/first-record, not mid-body.**
Perl `bam_isTruncated` scans the first ~10 `samtools view 2>&1` lines and dies on a bracketed
EOF/truncated message (gated on `\.bam$`). The Rust maps an I/O error during `read_header()` or
the first-record peek to `Truncated` (`pipeline.rs:294`), but a BAM truncated **mid-body**
(valid header + valid record 1, corrupt later BGZF block) surfaces as a plain `Io` error
*during streaming*, after partial output is written — whereas Perl could catch the EOF-absent
warning up front. This is an **error-path** detail (not byte-gated), and noodles does still
error on the corrupt block (it never silently truncates output), so data integrity is
preserved. Acceptable; document under SPEC §10.6 if not already. Severity Low.

**B-3 (Low, informational) — `extract_xm` non-`String` tag value → `None` → treated as absent.**
`extract_xm` (`pipeline.rs:253`) returns `None` for a non-`Value::String` XM. In SE this means
"kept" (matches absent-XM); in PE this means the die path. Perl's regex `XM:Z:(.+?)\s` only
ever matches a `Z`-typed string tag in text form, so a non-string XM can't occur in real
Bismark output. Faithful for all real data; flagging only for completeness.

### Efficiency

**B-4 (Low) — `qname_string` allocates only on the error path.** Fine. The hot path
(`qnames_match`, `read_fails`, `extract_xm`) is allocation-free and byte-oriented. `report.format()`
uses `String::with_capacity(256)` — good. No bottlenecks. The single-threaded + mimalloc choice
matches the sibling crates and SPEC §10.4. No action.

### Errors / robustness

**B-5 (Low) — `round_1dp` `.expect(...)` is provably unreachable** (a `{:.1}`-formatted finite
float always reparses). Inputs are `count/total*100` with `total ≥ minimum_count ≥ 1`, so finite
and non-NaN. Safe. No `unwrap`/`expect` on any real-data-reachable path; `#![forbid(unsafe_code)]`
in both `lib.rs` and `main.rs`. Good.

**B-6 (Low) — `eprintln!`/`eprint!` failures are ignored (`run`, `lib.rs:64`).** STDERR write
errors are silently dropped (the macros panic only on a poisoned lock, not on EPIPE — they use
`Stderr`'s `Write` impl which ignores errors). Perl `warn` likewise can't fail the run. Faithful;
no action.

### Structure / style

**B-7 (Nit) — `is_coordinate_sorted` (pipeline.rs:283) duplicates `bismark_io`'s
`check_not_coordinate_sorted` logic** (`read.rs:620`). The crate re-implements the `@HD SO:` ==
`coordinate` check locally because `bismark-io` only exposes it as part of its record reader
(which is intentionally bypassed here). Acceptable duplication given the raw-`RecordBuf` design;
not worth a shared-API change. The PE-only gating correctly matches Perl (Perl calls
`test_positional_sorting` only `if ($paired)`).

**B-8 (Nit) — module/`doc` comments are unusually thorough and accurate** — they cite exact
Perl line numbers and the load-bearing quirks. This is a strength; no change. Test names are
descriptive and the test matrix matches SPEC §12. The `.sam$` branch of Perl's emptiness gate is
correctly unreachable here (SAM rejected at the top `bam$` gate) and not implemented — correct.
