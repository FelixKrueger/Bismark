## CRITICAL

### C-1. The plan promises streaming `cmp` but reuses a harness that uses memory-buffering `diff` — at 10M this OOMs or silently changes method.
- **§3.1** is explicit: "via **streaming `LC_ALL=C cmp`** … O(1) memory; on mismatch, map the byte offset to a line window with `sed` — never re-`diff` the full stream." **§5** repeats "streaming `cmp` (O(1) memory)."
- But **§4 step 3** says the harness "reuses the `run_cell` skeleton from `phase4_minimap2_se_gate.sh`." That `run_cell` (verified, lines 65/74) compares with **`diff <(filter_sam "$pbam") <(filter_sam "$rbam") > …diff`**. `diff` is a **buffering** comparator — it must hold (at least) the LCS state / both line sets in memory. The same is true of `phase2{a,b}` harnesses (lines 58/63). At 10M records a decompressed SAM is on the order of **2–3 GB of text SE / 4–6 GB PE**; `diff` of two such streams is exactly the failure mode Phase 10 §3.4 documents: *"`diff <(samtools view …) <(…)` buffers both streams … exhausts memory."* The 32c/**256 G** cgroup makes an outright OOM unlikely for SE 10M, but PE 10M + the `--ambig_bam`/aux + `diff`'s superlinear behavior on a large *mismatch* (it tries to compute a minimal edit script over millions of differing lines) is a real risk; and even when it "works" it is slow and writes a multi-GB `.diff` file to `/var/tmp`.
- **The deeper risk is pass-while-wrong by method-substitution:** the plan *says* `cmp` but the reused skeleton *does* `diff`. If the implementer copy-pastes phase4's `run_cell` (the literal instruction in step 3) the gate silently runs `diff`, not the O(1) `cmp` the plan was reviewed on. The Phase-10 `phase10_subset_strict_gate.sh` already solved this correctly (`cmp_files()` using `cmp -s` + a *bounded* `sed` window on mismatch, lines 75–84). **Phase 5 must reuse the Phase-10 `cmp_files`/`compare_dirs` machinery, NOT the phase4 `diff` skeleton.** The plan names the wrong template.
- **Fix:** Change §4 step 3 to "reuse the **`phase10_subset_strict_gate.sh`** comparison core (`cmp_files`, `compare_dirs ordered`, the empty-BAM backstop, `n_sam_ordered`/`n_report`/`n_fq_ordered`)"; explicitly forbid `diff` on the full stream. Adapt that harness to be parameterized on `(backend, layout, library, genome)` and to set the per-backend invocation flag (`--hisat2` / `--minimap2` / Bowtie 2 default). This also *for free* inherits the Phase-10 **empty-BAM backstop** (lines 188–197), which is the truncation guard for probe #6 (below).

## IMPORTANT

### I-1. The 10M in-order `cmp` does NOT subsume Phase 10's truncation/count guard unless the harness asserts a non-empty, count-matched main BAM first.
Probe #6 asked whether a byte-`cmp` of `samtools view -h` *subsumes* count + header + RNAME-set "*if* it truly runs to completion." It does — but only conditionally. `cmp -s a b` returns **0 (PASS) when both files are identical, including both empty**. If a backend produced an empty-but-exit-0 BAM on *both* sides (e.g. a path/genome misconfiguration that makes both Perl and Rust align 0 reads, or a `samtools view` that errored to an empty stream under `set -uo pipefail` with the error swallowed by a pipe), the `cmp` would **PASS vacuously**. The Phase-10 harness explicitly guards this with the backstop at lines 188–197 ("refusing to pass vacuously"; assert `nsc>=1` and `perl_sc==rust_p1` counts). **The Phase 5 plan's §3.1/§8 does not mention this backstop at all.** Because the plan also drops Phase 10's B1.5 count-reconciliation as "subsumed by cmp," the *only* thing standing between a silent truncation and a green gate is a guard the plan never specifies. **Add the non-empty + count-equality backstop to §3.1 and V1/V2 explicitly** (it is nearly free — one `samtools view -c` per side).

### I-2. The non-dir/pbat cells on a directional 10M dataset prove almost nothing new — and the plan's honesty here is *partial*, not full.
Probe #4. The plan *does* surface the Phase-8 caveat ("~0 reads on the complementary strand," §3.2 / Assumptions §7). But it then lists **six** non-dir/pbat cells (`ht2_se_nondir`, `ht2_se_pbat`, `ht2_pe_nondir`, `ht2_pe_pbat`, `mm2_se_nondir`, `mm2_se_pbat`) in the headline matrix as if they were scale datapoints. From the Phase-4 GATE_OXY, at **1M** pbat landed **51 records** and non-dir **6,858**; scaled to 10M that's ~510 / ~69k — a *rounding error* relative to the ~8M directional records. So:
   - These cells are **dominated by the same dominant-strand reads** as the dir cell → they mostly re-prove the dir cell at 10M, plus the *routing* (which the 10k/1M per-phase gates already proved).
   - **Pass-while-wrong exposure:** a real non-dir/pbat *complementary-strand* bug that only manifests at scale would be masked, because there are almost no complementary-strand reads to exercise it. The plan asserts these cells "prove byte-identity-at-scale + routing, not new strand coverage" — which is honest about coverage but **understates that a complementary-strand-only bug is invisible here**.
   - The plan offers the **R1↔R2-swap pbat trick** (Phase 10 rev-2, harness line 216) for `ht2_pe_pbat` ("pbat via R1↔R2 swap if a real pbat signal is wanted") but leaves it **optional/conditional** ("else dominant-strand only"). For the *PE pbat* cell this is the only way to get a non-trivial CTOT/CTOB population. **Recommendation:** either (a) commit to the R1↔R2 swap for `ht2_pe_pbat` so at least one HISAT2 cell exercises a real complementary-strand population at 10M, or (b) explicitly demote the six near-empty non-dir/pbat cells to "routing regression at scale (low new signal)" in the matrix so the gate's actual coverage isn't overstated when the README later says "v1.x complete." Note the swap trick does **not** rescue *non-directional* (no analogous swap), so non-dir at scale remains genuinely thin — state that as accepted residual the way Phase 10 O5 did.

### I-3. The minimap2 in-order premise should be re-verified at 10M before the strict `cmp` relies on it — the spike only validated order at 10k, and the 1M precedent used `--parallel` chunking, not a 10M single-stream.
Probe #1. The strict streaming `cmp` is only valid for minimap2 if the *single*-instance (`--parallel 1`) output is in exact input order at 10M. Evidence: spike Q5 validated at **10k** ("spans ~3 minibatches at `-K 250K`"); the Phase-4 1M worker-invariance cell validated `-t 2` order-preservation but **under `--parallel 8`** (i.e. 8 chunks each ≤~125k reads). Neither is a *10M single `--parallel 1`* stream at `-K 250K` (≈ tens of minibatches feeding `-t 2` worker threads, where minibatch results are reordered to input order by minimap2's collector). minimap2's input-order guarantee is real (its multi-thread output reordering is by design), and the Bowtie 2 `--reorder` analog makes this very likely — but the plan *bets the strict `cmp`* on it without a cheap pre-check. **Add a one-line guard in the harness:** before the strict `cmp`, assert `samtools view rust_p1.bam | cut -f1` is in the *same order* as the input FastQ read IDs for `mm2_se_dir` (or simply rely on the Phase-10-style `cmp` itself failing loudly with a bounded diagnostic window — which *is* fail-loud, so this is belt-and-suspenders). At minimum, the plan should state that an out-of-order `mm2` cell at 10M is a **plausible false-FAIL** and the diagnosis recipe (not a method-downgrade to sort) is the response. The plan currently asserts in-order as settled fact.

### I-4. Folding the gate + README bump into already-green #950: the README "v1.x complete" line must not land before the gate PASSES, and CI re-run scope is understated.
Probe #7. #950 is currently green/MERGEABLE with only the minimap2 SE code. Phase 5 adds (a) gate docs/harness (non-code), (b) a `rust/README.md` bump claiming "v1.x complete + byte-identical across all 3 backends." The plan's §4 step 6 correctly sequences "On PASS → bump README … push to #950," and step 7 says "On FAIL: stop, do not auto-fix." Good. But two gaps:
   - The plan does not state that **the README claim is only valid after the *combined 10M* gate passes** — not after the *per-phase* gates. If an implementer, eager to close the epic, pushes the README bump *with* the harness *before* running the (multi-hour) HISAT2 cells, #950 goes green on CI (CI doesn't run the oxy gate — EPIC §2 OUT: "HISAT2/minimap2 in the perl-oracle CI gate" is a follow-up) and the merge could happen on a claim the gate never substantiated. **Make explicit: do not push the README "complete" line until `GATE_OXY.md` records ALL cells PASS.** (This matches `~/.claude/CLAUDE.md` verify-after-implement, but the plan should nail it down given the fold.)
   - CI re-run scope: pushing non-`rust/` plan docs + a `rust/README.md` text bump re-triggers the full CI matrix (fmt/clippy/test/perl-oracle). The README is a doc-only change so it's cheap, but the plan should note the `cargo fmt`/clippy doc-bullet traps don't apply to README markdown — *however* if the bump touches any `///` rustdoc (it shouldn't) the `doc_lazy_continuation` gate bites. Low risk; worth a one-liner so the implementer doesn't accidentally edit a docstring.

## OPTIONAL

### O-1. Branch-name inconsistency between EPIC.md and this plan (cosmetic but confusing).
`EPIC.md` line 4 says the epic branch is `rust/aligner-v1x`; the Phase-5 plan (and PROGRESS, and reality — `git branch` shows `rust/aligner-mm2` is the checked-out branch carrying `21bac5d`) uses **`rust/aligner-mm2`**. The Phase-4 plan explains the divergence (the local `rust/aligner-v1x` diverged at the #949 squash → Phase 4 cut a fresh `rust/aligner-mm2`). The Phase-5 plan is *internally* correct and even names the commit, so this is not a blocker — but `EPIC.md` is now stale. Recommend a one-line note in EPIC.md (or PROGRESS) that the live branch is `rust/aligner-mm2`, so the "fresh-branch PR" step in §4.6 doesn't get pushed off the wrong base.

### O-2. Runtime realism for single-core HISAT2 at 10M — the plan says "hours/cell" but gives no estimate or budget; quantify so a recycle-window risk is sized.
Probe #7. Single-core HISAT2 on 10M human WGBS reads × 2 instances (CT+GA) is the runtime driver. The plan accepts this but never estimates it. The faithful Phase 10 used *84M* but with `--parallel 8`. Here HISAT2 is **forced single-core**, so the wall is genuinely serial. With ~8 HISAT2 cells (SE dir/non-dir/pbat + PE dir/non-dir/pbat + mouse PE), at "hours/cell," the full sequential matrix could be **a day-plus** — well beyond a typical ephemeral-pod lifetime, raising recycle exposure (memory note: "a pod RECYCLE … wipes the ephemeral /var/tmp + kills detached jobs"). The plan's per-cell off-box capture + idempotent re-run is the right mitigation, but **the plan should (a) give a rough per-cell wall estimate from the 1M per-phase timings ×10, and (b) consider running the *Perl* single-core side once and reusing it** — note the gate compares Rust-sc vs Perl-sc per cell, and the dir/non-dir/pbat cells share NO Perl run (different argv), so there's no reuse there, unlike Phase-10 Gate A which reused one Perl-sc across three legs. So Phase 5's serial cost is genuinely ~2× the number of cells in long HISAT2 runs. Worth sizing before committing to "keep the full ~12-cell matrix" (OQ-5b).

### O-3. `--ambig_bam` / aux handling for the minimap2 and Bowtie 2 cells is asserted but the cell argv in §3.2 doesn't show the aux flags.
§3.1 says aux (`--unmapped`/`--ambiguous`/`--ambig_bam`) is compared (decompressed, FastQ record-ized, `--ambig_bam` via `samtools view`), and the Phase-10 harness passes those flags on every run. But the Phase-5 §3.2 cell table doesn't show whether all 12 cells pass `--unmapped --ambiguous --ambig_bam`. The Phase-4 SE harness did **not** pass them (it only compared the main BAM + report). Recommend the plan state explicitly which cells carry the aux flags (Phase-10 ran them on all; minimal cost) so the aux-comparison machinery in §3.1 isn't dead code, and so a `--ambig_bam` HISAT2 PE path (the one raw-passthrough path, per the 2b harness comment) is exercised at 10M.

## Summary of what the plan got RIGHT (so changes are scoped, not a rewrite)
- The **per-backend gate-shape table (§2)** is correct and is the load-bearing insight: HISAT2 single-core-strict (no worker leg, no content shortcut — verified against `config.rs`), minimap2/Bowtie 2 get the cheap worker/anchor legs.
- The **10M-strict-in-order → `cmp`** choice (vs Phase-10's 84M content-multiset) is the *right* simplification **for the in-order backends** — provided C-1/I-1/I-3 are honored.
- **OQ-5a is empirically TRUE** (I verified the mouse `.ht2`/`.bt2` on oxy live); the surface-don't-auto-build fallback is safe and now moot.
- The **faithful-port circularity / Perl-is-the-oracle** framing is honest (§10 risk d).
- The **no-production-code / zero-regression-surface** claim is correct — the binary is `21bac5d`, unchanged.

## Highest-risk finding
**C-1:** the plan instructs reuse of the `phase4` `diff`-based `run_cell` while *claiming* O(1) streaming `cmp`. At 10M (esp. PE + on a real mismatch) `diff` buffers/blows up, and — worse — the gate silently runs a different, weaker-by-omission method than the one being reviewed. Must reuse the Phase-10 `cmp_files`/`compare_dirs ordered` core (which also brings the empty-BAM truncation backstop that I-1 needs) before implementation.
