## 1. Logic review

### 1.1 Parallel-array companion logic + line-1256 `undef`-reset — **CORRECT**
The five parallel arrays (`@alignment_reports` + 4 companions), the `while (@alignment_reports){ shift … }` loop, and the per-iteration independence are described faithfully (SPEC §2.1–§2.2). I confirmed line 1256 (`$dedup_report = $splitting_report = $mbias_report = $nucleotide_coverage_report = undef;`) resets the **explicit** companion vars at the end of each `foreach my $aln` iteration, so an explicit `--dedup_report X` applies only to the first alignment report and subsequent reports fall back to auto-detect (SPEC §2.2 bullet 3, §8.11). This is correctly flagged as subtle and "mirror exactly."

One nuance the SPEC does not state but should: the reset is in the **`process_commandline` glob/build loop** (lines 1134–1257), *not* in the main `while (@alignment_reports)` processing loop (lines 34–158). The Rust port must reproduce the reset **while building the parallel arrays**, not while consuming them. The SPEC's prose ("a second alignment report falls back to auto-detection") is right but a careless implementer could put the reset in the wrong loop. Minor wording fix.

### 1.2 The five parsers — **CORRECT** (verified line-by-line)
- **PE-vs-SE detection** (SPEC §2.7a): the eight branch pairs (total/unique/no-aln/multiple/no-genomic) match Perl lines 215–273. The human-label `*_text` strings are captured by the SPEC implicitly; **the exact label strings are NOT reproduced in the SPEC** (e.g. `'Paired-end alignments with a unique best hit'` vs `'Single-end alignments with a unique best hit'`, line 236/241; `'Pairs without alignments under any condition'` vs `'Sequences without alignments under any condition'`, 247/252; etc.). These strings are injected into `{{unique_seqs_text}}` / `{{no_alignments_text}}` / `{{multiple_alignments_text}}` / `{{sequences_analysed_in_total}}` and are **byte-load-bearing**. The SPEC should lift them verbatim into the implementation contract (the PLAN can carry them, but the SPEC currently leaves them as "(which also sets the human label `*_text`)" without the bytes). *Important — see Action Items.*
- **Alternate unmeth phrasing** (SPEC §2.7a vs §2.7c): correctly captured. Alignment parser accepts **both** `Total unmethylated C's in … context:` **and** `Total C to T conversions in … context:` (Perl lines 298–313, `or`-joined). Splitting parser accepts **only** `Total C to T conversions in … context:` (lines 745–760). Also correctly noted: alignment uses `C methylated in Unknown context (CN or CHN):` (line 330) while splitting uses `C methylated in Unknown context:` (line 777) — the `(CN or CHN)` suffix differs. Verified. Good catch by the SPEC author; these are exactly the kind of near-duplicate that produces silent divergence.
- **Strand-origin patterns** (SPEC §2.7a): PE `^CT/GA/CT:` / SE `^CT/CT:` (OT); PE `^GA/CT/CT:` / SE `^GA/CT:` (CTOT); PE `^GA/CT/GA:` / SE `^GA/GA:` (CTOB); PE `^CT/GA/GA:` / SE `^CT/GA:` (OB). Verified against Perl lines 339–372. **Subtle ordering hazard the SPEC under-states:** these are `elsif` branches, so order matters — e.g. SE `^CT/CT:` (line 343) is tested *before* nothing that would shadow it, but note `^CT/GA:` (SE OB, line 370) must not be reached for a PE line `CT/GA/CT:` because the PE branch `^CT/GA/CT:` (line 339) is anchored and tested first. I verified the anchoring makes them mutually exclusive (a PE line `CT/GA/CT:` does **not** match `^CT/GA:` only because `^CT/GA/CT:` wins the earlier `elsif`; but a *standalone* `CT/GA:` line cannot be confused). The Rust port must preserve **first-match-wins `elsif` order**, not a hashmap of independent prefix tests, or a PE line could match an SE prefix. SPEC §8.10 says "verify the SE patterns don't accidentally match PE lines (they don't — distinct anchoring)" — this is **only true because of `elsif` precedence**, not pure anchoring (`^CT/GA:` *is* a prefix of `^CT/GA/CT:`). The SPEC's parenthetical "distinct anchoring" is slightly wrong reasoning for a correct conclusion. *Important to state the real reason so the implementer preserves branch order.*
- **Alignment all-or-nothing fill gate** (SPEC §2.7a): `defined $unique and … $no_aln … $multiple … $no_genomic … $total_seqs` (Perl line 378). Verified. Correctly noted that on failure the placeholders survive (`warn "Am I missing something?"`, line 502).
- **Dedup leftover fallback + `\s.*` trims** (SPEC §2.7b): `$dups =~ s/\s.*//` (line 530), `$diff_pos =~ s/\s.*//` (line 535), and the fallback `$leftover = $total_seqs - $dups` (lines 544–548). Verified. The integer-subtraction emit is correct (Perl numeric `-` then stringify). Note: the SPEC says "integer subtraction" — Perl `-` on `"100"` and `"5"` yields `95` and stringifies to `"95"`; if either operand had a fractional part it would be float, but dedup counts are always integers, so this is safe. Fine.
- **Nucleotide fixed 20-key order + header validation + commented-out log2** (SPEC §2.7e): the hardcoded order `A,T,C,G,AC,CA,TC,CT,CC,CG,GC,GG,AG,GA,TG,GT,TT,TA,AT,AA` matches Perl line 632 exactly. Header validation checks **only col 3 (`percent sample`) and col 5 (`percent genomic`)** — col 1 (`$element`) is **not** validated (Perl lines 587–601). SPEC §2.7e correctly omits a col-1 check. The log2 ratio (`$logratio = sprintf("%.2f", …)`) is commented out (lines 657–660) and its injection `{{nuc_minmax}}` is also commented out (line 692) → no float reaches output. Verified. This is the lynchpin of the "no numeric reformatting" claim (§2.7 / §8.7) and it holds.

### 1.3 Asset-injection order vs placeholder-fill order — **CORRECT**
SPEC §2.3 lists the mutation order: plot.ly → bismark.logo → bioinf.logo → timestamp → alignment → dedup → splitting → mbias → nucleotide. Verified against Perl lines 59–156. I confirmed `getLoggingTime` (line 85) runs **before** `read_alignment_report` (line 90), so `{{date}}`/`{{time}}` are filled first. ### 1.4 Greedy/dotall section-deletion semantics — **CORRECT, with a sharpening**
SPEC §2.4 / §8.3: `s/\{\{marker\}\}.*\{\{marker\}\}//s` deletes first→last inclusive; each marker occurs exactly 2× (I verified all 8 markers occur exactly twice via grep). The proposed Rust impl ("find first index of marker and last index of (marker + its length), splice out the span") is correct **provided** "last index" means the index of the *last* occurrence's *start* plus the marker length — i.e. `rfind(marker)` then extend by `marker.len()`. The SPEC wording "last index of (marker + its length)" is ambiguous; recommend the PLAN spell it as: `start = doc.find(MARKER)`, `end = doc.rfind(MARKER) + MARKER.len()`, `doc.replace_range(start..end, "")`. Because each marker appears exactly twice, `find`/`rfind` land on the two distinct occurrences. **One caveat:** the M-bias R2 SE-deletion in Perl wraps the pattern in a capture group `s/(\{\{mbias_r2_section\}\}.*\{\{mbias_r2_section\}\})//s` (line 127) — the capture is only for the commented-out `warn "Deleting: $1"`; functionally identical to the un-grouped form. SPEC §2.7d does not mention the capture group, which is fine (it's inert), but worth a one-line note so a reviewer diffing the Perl isn't confused.

### 1.5 M-bias R1/R2 SE-vs-PE deletion asymmetry — **CORRECT but the SPEC misses a consequential side-effect**
SPEC §2.3 step 9 / §2.7d / §8.4 correctly describe: present → delete `{{mbias_r1_section}}` markers + fill; SE → delete the *whole* `{{mbias_r2_section}}` block; PE → delete only the R2 markers; absent → delete *both* R1 and R2 blocks.

**GAP (Critical for byte-identity coverage):** The `{{mbias_r1_section}}` markers are at template **lines 465 & 480**; the `{{mbias_r2_section}}` markers at **lines 483 & 496**. But the actual **data placeholders** `{{mbias1_*_*_*}}` live in the Read-1 *script block* at **lines 843–907**, and `{{mbias2_*_*_*}}` in the Read-2 script block at **lines 997–1060** — i.e. **OUTSIDE the deletable section spans**. Consequences the SPEC does not state:
1. **M-bias absent:** the else-branch (Perl lines 134–138) deletes only the two *section marker spans* and never calls `read_mbias_report`. Therefore the 12 `{{mbias1_*}}` placeholders (843–907) **and** the 12 `{{mbias2_*}}` placeholders (997–1060) are **never filled** and **never deleted** → they **survive as literal `{{…}}` text in the output**. This is a real, contractual output state that the Rust port must reproduce byte-for-byte, yet SPEC §5.4 ("unfilled placeholders are a real output state") only frames this around the *fill-gate* failures of alignment/splitting/nuc/dedup — it never says the M-bias script placeholders survive when M-bias is **absent**. An implementer who "tidies up" by deleting or zeroing these would diverge.
2. **M-bias SE present:** `read_mbias_report` fills `{{mbias1_*}}` always, but `{{mbias2_*}}` only `if (%mbias_2)` (Perl line 977), which is false for SE. So the 12 `{{mbias2_*}}` script placeholders **survive as literal text** in SE output, even though the visible R2 *section* (483–496) is excised. SPEC §2.7d notes the section deletion but not that the R2 *script* placeholders survive.
3. The dead `{{bm_mbias_2}}` substitution (Perl line 1016) fires in the SE branch of `read_mbias_report` (the `else` of `if (%mbias_2)`) and targets a non-existent placeholder → no-op (SPEC §2.7d / §4.7 / §8 correct).

**This is the single most important addition for rev 1.** It directly governs which bytes appear in the most common real-world cases (SE samples; samples processed without `--mbias`). It also dictates a mandatory fixture (see §4).

### 1.6 `read_report_template` line-normalization byte contract — **MOSTLY CORRECT; one edge bug + one nuance**
SPEC §2.6 / §8.2: per-line `chomp` + `s/\r//g` + append `\n`; document fully LF-normalized; `$doc` always ends in `\n`; mid-line `\r` removed. I verified all of this empirically:
- Mid-line CR + CRLF terminator `"ab\rcd\r\nef\n"` → Perl yields `61 62 63 64 0a 65 66 0a` = `"abcd\nef\n"`. **Confirmed** (`chomp` strips only the trailing record-sep `\n`; `s/\r//g` then strips the surviving CR before the old `\n` *and* the mid-line CR).
- No-trailing-newline content `"line1\nline2"` (this is the real shape of `plot.ly`, which I confirmed ends in `>` with no `\n`) → Perl yields `"line1\nline2\n"` (appends `\n` to the last line). **Confirmed.** The SPEC's proposed Rust normalizer (split on `\n`, drop a trailing empty element only if content ended in `\n`, strip `\r` per piece, rejoin with `\n`, append final `\n`) reproduces this exactly. Good.
- Trailing-newline content `"a\n"` → Perl `"a\n"`; the proposed normalizer also yields `"a\n"`. **Confirmed.**

**EDGE BUG (Important):** For an **empty file**, Perl's `while(<DOC>)` body never executes, so `$doc` stays `""` (length 0) — it does **NOT** get a trailing `\n`. But the SPEC's blanket claim "`$doc` always ends in `\n` regardless of whether the source file did" (§2.6) is false for empty input, and the proposed normalizer would produce `"\n"` for `""` (split `""` → `[""]`, content didn't end in `\n` so keep, rejoin → `""`, append `\n` → `"\n"`). The four real assets are all non-empty, so this never bites in production — **but** the §8.2 normalizer is described as a general helper and the PLAN's unit tests should pin the empty-input case to `""` (guard: if input is empty, return `""`). Recommend the SPEC add: "empty asset → empty string (no appended `\n`); the four shipped assets are non-empty so this is a unit-test guard, not a runtime path."

**Nuance:** the SPEC says use a faithful helper rather than `str::lines()` because `lines()` only strips a *trailing* `\r`. Correct, but note `lines()` *also* drops a final empty line and does not append a terminator — so the "split on `\n` + explicit handling" approach is the right call. The SPEC already says this.

### 1.7 Exact bytes of the 3 Unknown-context `<tr>` inject snippets — **CORRECT (verified with `cat -A`)**
SPEC §2.7a / §8.5 claims: header `<tr>` line = 5 spaces; `<th>` line = 32 spaces; `<td>` line = 4 spaces + 4 tabs; `</tr>` line = 4 spaces + 3 tabs. I verified against Perl lines 433–444 with `cat -A`:

- Line 434 (`<th>`): 32 leading spaces before `<th>` ✓ (confirmed length = 32)
- Line 435 (`<td>`): `····\t\t\t\t<td>` → **4 spaces + 4 tabs** ✓
- Line 436 (`</tr>`): `····\t\t\t</tr>` → **4 spaces + 3 tabs** ✓

The snippets are multi-line heredoc-style double-quoted strings with embedded literal newlines; the **same** three snippets appear in both `read_alignment_report` (433–444) and `read_splitting_report` (817–828) with identical whitespace. The splitting version injects into `{{*_unknown_splitting}}` placeholders (835–837). The SPEC's "(same byte layout as (a))" for splitting (§2.7c) is correct. **The SPEC's byte description is accurate — this is the highest-risk-to-typo item and the author got it right.** Strong recommendation: ship these as `const` `&str` lifted verbatim from the Perl source and assert byte-equality in a unit test; do **not** hand-retype them.

### 1.8 Output filename derivation — **CORRECT**
SPEC §2.5 matches Perl lines 42–56: strip dir (`s/^.*\///`), strip `.txt` (`s/\.txt$//`), append `.html`, then `-o` override (verbatim, single-report-only), then prefix `$output_dir`. - **"Assets contain no live `{{…}}` token or replacement metacharacter" (§8.13):** I confirmed `grep -c '{{'` = **0** for all three assets (`plot.ly`, `bismark.logo`, `bioinf.logo`). - **"Zero numeric reformatting except %-strip, dedup `\s.*`-trim, integer leftover" (§2.7 / §8.7):** Verified. The only `sprintf` float (`%.2f` log2 ratio) is commented out *and* its injection is commented out. The `localtime` `sprintf("%02d…")` is the only live `sprintf` reaching output, hence "sprintf parity collapses to the timestamp line." Sound.
- **Replacement-string safety also covers captured VALUES, not just assets:** I empirically tested `s/\{\{filename\}\}/$input_filename/g` with `$input_filename = 'foo$1bar'` → output `foo$1bar` (the `$1` inserted **literally**, not interpreted). So Rust `.replace(marker, &value)` is byte-equivalent for value injection too. SPEC §8.13 frames this only around assets; it actually generalizes — worth a sentence so reviewers don't worry about a report field containing `$`/`\`.
- **Stale-reference diagnosis (§7 / §8.1):** Confirmed. `bismark_bt2_PE_report.html` line 152 = `Data processed at 11:25 on 2018-08-16` (HH:MM, **no seconds**) and line 1091 footer = `version v0.19.1`. Current `getLoggingTime` emits `HH:MM:SS` (line 170). The SPEC is **correct** that this file must NOT be the oracle. (Excellent catch; this is the kind of trap that silently poisons a golden.)
- **Glob sort order (§3 / §4.8):** Confirmed Perl `<*E_report.txt>` returns lexically sorted results (`a_PE_report.txt, b_PE_report.txt, c_SE_report.txt`). The SPEC's "low-stakes" assessment is right: each alignment report yields an independent file, and multiple companion matches `die`, so order affects only STDOUT and the (cosmetic) processing sequence — never output bytes of any single file. Reproducing a deterministic C-locale sort (as genomeprep §8.1 does) is the safe convention.

### 2.2 Ambiguous / under-specified assumptions to flag
- **`defined`-vs-truthiness companion asymmetry (NOT in the SPEC):** Perl uses `if ($dedup_report)`, `if ($splitting_report)`, `if ($mbias_report)` (truthiness, lines 1142/1202/1229) but `if (defined $nucleotide_coverage_report)` (line 1171). Consequences: `--nucleotide_report ''` is treated as **explicit** (defined) → not `'none'` → the empty string is pushed as the report path (and later `if ($nuc_report)` at line 141 is falsy, so it's skipped — net: behaves like absent, but via a different branch). For the other three, `''` or the literal string `'0'` is **falsy** → they fall through to **auto-detect**. So `--dedup_report 0` would *auto-detect* rather than use a file literally named `0`. clap will not naturally reproduce this Perl truthiness quirk. **Low-stakes** (these are pathological inputs), but the SPEC's §2.2 table presents all four companions uniformly ("`none` → skip; auto-detect if omitted") and should note the asymmetry, or explicitly declare it an accepted divergence. *Important to document, optional to replicate.*
- **Exit codes (§6.1 is muddled):** Both `--help`/`--man` (Perl line 1073–1076) **and** the no-`--alignment_report`/0-match path (lines 1116–1120) call `print_helpfile()`, which **itself** ends with `exit 1` (line 1314). The bare `exit;` statements at lines 1075 and 1119 are **unreachable dead code**. So **both paths exit with code 1**, not 0. SPEC §6.1 says "Perl `print_helpfile` exits 1, but the no-report path calls `print_helpfile()` after a `warn` and then `exit` separately" — the "and then `exit` separately" implies a second, reachable exit, which is wrong (it's never reached). Net effect for the port: if mirroring exit codes matters, `--help` should exit **1** (Perl-faithful), which is unusual for Rust/clap (clap exits 0 on `--help`). The SPEC lists `--help`/`--version` text as NOT gated (§4.3), and presumably exit codes likewise — but it should **explicitly state** whether the `--help`-exits-1 quirk is reproduced or accepted as a divergence. The `--version` path (line 1078–1090) uses a **reachable** bare `exit` → **exit 0**. So Perl: `--help`/`--man` → 1, `--version` → 0, no-report → 1. *Important — clarify the intended exit-code contract.*
- **`SOURCE_DATE_EPOCH` vs hidden `--_test_timestamp` (§4.5 / §7):** Left as "confirm the choice in review." My recommendation in §5 below. Flagging that this is still an **open decision**, not a settled assumption.
- **Crate name `bismark-report` vs `bismark2report` (§3):** The SPEC itself flags the inconsistency with the sibling convention (genomeprep crate = `bismark-genome-preparation` = full Perl name; by that rule this crate would be `bismark2report`). Open decision. See §5.

## 3. Efficiency analysis

- **Binary bloat from `include_str!`-ing ~3 MB `plot.ly`:** The SPEC (§3 / §3 reuse table) accepts ~3 MB of binary growth to keep a self-contained binary. This is the right trade-off and consistent with the rewrite ethos; the alternative (read from `$RealBin/plotly/`) reintroduces a runtime path dependency the other ports removed. `include_str!` validates UTF-8 at compile time — confirm `plot.ly` is valid UTF-8 (it is JS/minified text; base64 logos are ASCII). If any asset were *not* valid UTF-8, `include_str!` would fail to compile and you'd need `include_bytes!` + `from_utf8`. Given the assets are text, `include_str!` is fine; worth a one-line note that the embed is UTF-8-validated at build time.
- **Report slurp vs stream:** Reports are tiny (KBs); reading line-by-line (matching Perl's `while(<FH>)`) is both faithful and trivially fast. No concern. The 3 MB `$doc` string with ~30 sequential `.replace()` passes is O(N) each ≈ 30×3 MB ≈ 90 MB of transient copying per report — negligible for a per-sample tool. No optimization needed; do **not** prematurely switch to a single-pass templating engine, as that would risk byte-divergence vs Perl's sequential `s///g`.
- **`.replace()` vs `s///g` semantics:** Rust `str::replace` replaces **all non-overlapping** occurrences left-to-right, single-pass — matching Perl `s///g`. Confirmed equivalent for the value/marker substitutions. For the greedy section deletion, `replace` is the wrong tool (it would delete *both* markers and leave the content); the `find`/`rfind`/`replace_range` splice is correct (SPEC §2.4). The SPEC distinguishes these correctly.

## 4. Validation sufficiency

### 4.1 Is the acceptance gate enough?
The **Perl-script-as-primary-oracle** approach (§7, mirroring genomeprep/methcons) plus committed hand-checked goldens is the right backbone and is sufficient *if* the fixture matrix covers the byte-divergence risks. The timestamp-line normalization in **both** outputs (§7.1) is **sound and does not mask real divergence**, because: (a) the timestamp appears on exactly **one** template line (line 146, verified — `{{time}}`/`{{date}}` occur only there); (b) `{{time}}`/`{{date}}` are filled by `getLoggingTime` with a fixed `sprintf` format; (c) normalizing `HH:MM:SS` and `YYYY-MM-DD` to fixed tokens on that one line in both files leaves **every other byte** under comparison. The only residual risk is if the normalization regex is too greedy and matches more than the intended line — recommend anchoring the normalization to the full literal `Data processed at \d\d:\d\d:\d\d on \d\d\d\d-\d\d-\d\d` so it cannot accidentally rewrite a data value that happens to look like a time. *Important: pin the normalization regex precisely in the PLAN.*

### 4.2 Edge cases that LACK a fixture in the current §7.2 matrix
The §7.2 matrix is good (PE/SE; with/without each of 4 optionals; Unknown-context present/absent; M-bias SE vs PE; `--dir`/`-o`; multi-report; `none` skips). Gaps:
1. **M-bias-absent → surviving `{{mbias1_*}}`/`{{mbias2_*}}` script placeholders** (the §1.5 GAP). The matrix has "without each optional report," which *includes* M-bias-absent — but the SPEC must explicitly state the **expected** output is that the 24 script placeholders survive as literal text, and the golden must capture exactly that. Without calling it out, an implementer/reviewer might "fix" it. **Add an explicit assertion.**
2. **M-bias SE → surviving `{{mbias2_*}}` placeholders** (the §1.5 SE sub-case). The matrix has "M-bias SE (R1 only)" but does not state the R2 script placeholders survive. **Add an explicit assertion.**
3. **Fill-gate-failure fixtures** (§5.4): the matrix does not include a *deliberately incomplete* alignment report (missing one of the 5 gate fields), splitting report (missing one of 6), dedup (missing one of 4), or nucleotide (`looksOK` failure / missing a key). These are the cases where the SPEC's "placeholders survive" contract (§5.4) is exercised. **At least one fixture per parser should trip the gate** and the golden should show literal `{{…}}` survivors. Highest value: a nucleotide report missing a key (the issue-#711 amplicon case, §8.9) — verify that the missing key's percentages render `0` and counts/coverage render the **empty string** (Perl undef-in-replacement: `$doc =~ s/…/$counts_obs/g` with `$counts_obs` undef inserts `''` and warns under `use warnings`). I confirmed the Perl logic: for a key in the fixed list but absent from `%nucs`, `$nuc_obs`/`$nuc_exp` default to `0` (lines 641–646) but `$counts_obs`/`$counts_exp`/`$cov` stay **undef** → inserted as empty string. **This is a real divergence risk** (a naive Rust impl might insert `"0"` or leave the placeholder); needs a fixture. *Important.*
4. **Dedup leftover-fallback path:** a dedup report **without** the `Total count of deduplicated leftover sequences:` line (so `$leftover = $total - $dups`). The matrix doesn't distinguish "leftover present" vs "leftover computed." **Add a fixture** for the fallback (older dedup reports lacked that line). *Important.*
5. **Companion-flag reset across multiple reports (§8.11):** the multi-report matrix entry should include a run with an **explicit** `--dedup_report X` *and* 2 alignment reports, asserting that report 1 uses `X` and report 2 auto-detects (or gets `''`). This exercises the line-1256 reset. *Important — it's the most counter-intuitive behavior.*
6. **`>1` companion match → `die`:** a fixture directory with 2 files matching `<basename>*deduplication_report.txt` should produce a nonzero exit / error. The matrix mentions "`>1` basename match → error" in §6.1 but no fixture is listed. *Optional (it's an error path, low byte-risk).*

### 4.3 Could plot.ly / a logo contain a live `{{…}}` or replacement metacharacter? — **ANSWERED: NO.**
Empirically confirmed (§2.1 above): `grep -c '{{'` = 0 across all three assets, and `s/pat/$var/` inserts variable contents literally. So neither a stray `{{token}}` in plotly.js nor a `$`/`\` in a value can corrupt output. Spike A (§8 / §8.13) is therefore **confirmatory, not blocking** — the SPEC's call ("none blocks the SPEC") is correct, and you can even drop the "confirm plot.ly contains no live `{{…}}`" sub-task since I've verified it (or keep it as a regression guard against future plotly.js upgrades, which is actually the more useful framing — see §5).

### 4.4 Is normalizing the timestamp in BOTH outputs sound? — **YES** (see §4.1). The one caveat is regex precision; otherwise it cannot mask a real divergence because the timestamp is confined to a single, uniquely-phrased line.

## 5. Alternatives & trade-offs

1. **Timestamp determinism mechanism (§4.5 / §7).** Three options:
   - **`SOURCE_DATE_EPOCH`** (reproducible-builds convention): clean, no public CLI surface, well-understood, and lets the **same env var** pin *both* the Rust run and (with a tiny wrapper) make goldens stable. **My recommendation.** It also signals "reproducible build" intent. Downside: needs the golden-generation harness to set it.
   - **Hidden `--_test_timestamp`:** explicit and self-documenting in tests, but adds a (hidden) CLI flag that must be kept out of `--help` and is a public-ish surface. Slightly more code.
   - **Just normalize at compare-time and never pin the source:** simplest, but then committed goldens are *not* byte-stable (they'd contain whatever timestamp was current at generation), which forces the test harness to normalize the golden too — workable but means the golden isn't a literal "expected bytes" file. The SPEC already plans to normalize at the Perl-vs-Rust gate; using `SOURCE_DATE_EPOCH` additionally for the *committed* goldens gives you a genuinely byte-stable checked-in expectation. **Prefer `SOURCE_DATE_EPOCH` for goldens + normalize-at-compare for the Perl oracle.** This is exactly the SPEC's leaning; endorse it explicitly in rev 1.
2. **Crate name `bismark-report` vs `bismark2report` (§3).** The sibling rule (full Perl name as crate name) argues for `bismark2report`. But Cargo crate names with digits-then-letters are legal and `bismark2report` reads oddly as a crate. The **binary** name `bismark2report_rs` is settled and correct (drop-in). For the **crate**, I'd lean to whatever keeps `rust/Cargo.toml` consistent — and the sibling precedent (`bismark-genome-preparation`, `bismark-coverage2cytosine`) actually *mixes* styles (`coverage2cytosine` keeps the digit; `genome-preparation` hyphenates). Given `bismark-coverage2cytosine` exists, the strict analog is **`bismark-bismark2report`** (silly) or **`bismark2report`**. I'd actually recommend **`bismark-report`** (the SPEC's proposal) as the cleanest, and just note the deviation from the "full Perl name" rule in the divergences list. Low-stakes; pick one and move on. The SPEC correctly flags it for review.
3. **Asset shipping.** `include_str!` is the right default. An alternative worth a sentence: gate the embed behind a build that *verifies* the asset's byte-length/hash at compile time (a `const_assert` on `EMBEDDED.len()`), so a future plotly.js bump that introduces a `{{` token or changes bytes is caught by a test rather than silently shipping a corrupt report. This converts Spike A's one-off check into a permanent regression guard. Recommend folding into the PLAN as a unit test: assert no `{{` substring in any embedded asset.
4. **Section deletion via regex crate vs manual splice.** The SPEC chooses manual `find`/`rfind` splice (no regex dep for deletion). Fine and avoids a regex engine's greedy/dotall config pitfalls. Keep it manual. (A `regex` dep with `(?s)` dotall would also work but is heavier and the manual splice is provably equivalent given the 2×-marker invariant.)
